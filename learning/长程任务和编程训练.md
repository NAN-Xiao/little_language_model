## 一、整体训练路径概览

```
基础 LLM（通用语料预训练）
    │
    ▼
┌─────────────────────────────────────────┐
│  阶段1: 代码持续预训练 (Continual Pre-training) │
│  用海量代码数据继续训练基础模型              │
│  → 模型"学会"编程语法、API、代码结构         │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│  阶段2: 代码指令微调 (Code SFT)              │
│  用 (问题 → 代码解答) 的指令对训练          │
│  → 模型"学会"按指令生成代码                  │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐

└─────────────────────────────────────────┘
    │
    ▼

    │
    ▼
┌─────────────────────────────────────────┐
│  阶段4: 长程任务适配 (Long-horizon Adaptation) │

    │
    ▼

```

## 二、重要前提：在已有模型上继续训练

**不是从头训练！** 所有阶段都是在前一阶段模型的基础上继续训练。

### 真实案例：Qwen2.5 → Qwen2.5-Coder

```
Qwen2.5 (通用基础模型)
    │  ↓ 用代码数据继续训练
    ▼
Qwen2.5-Coder (代码专用模型)
    │  ↓ 用代码指令数据微调
    ▼
Qwen2.5-Coder-Instruct (能对话的代码模型)
    │  ↓ 用强化学习优化
    ▼
Qwen2.5-Coder-RL (高质量代码生成模型)
```

### 成本对比

| 方式 | 数据量 | 计算成本 | 说明 |
|------|--------|---------|------|
| **从头训练** | 数万亿 token | 数百万美元 | GPT-4、Claude 的第一次训练 |
| **继续训练** | 数千亿 token | 数十万美元 | 在已有模型上"深造" |
| **微调** | 百万级样本 | 数千美元 | 在已有模型上"精修" |

### 本项目的训练入口

```bash
# 通用预训练（从头，或加载已有 checkpoint 继续）
python __main__.py train

# 代码持续预训练：加载通用模型，换代码数据集继续训练
python __main__.py train --resume-from checkpoints/xxx/best_model.pt

# 代码 SFT：加载代码模型，用指令对微调
python __main__.py finetune train --base-checkpoint checkpoints/xxx/best_model.pt --data-path code_instructions.jsonl
```

### 灾难性遗忘与防止

在代码数据上训练太久，模型会"忘记"通用语言知识。**解法：**

- **低学习率**：继续训练用 `1e-4`（预训练是 `3e-4`），更新幅度小
- **混合数据**：代码数据占 85%，通用语料占 15%
- **LoRA 微调**：只训练少量参数（A/B 矩阵），大部分冻结

```python
train_config = {
    "lr": 1e-4,
    "code_data_ratio": 0.85,
    "general_data_ratio": 0.15,
}
```

### 3.1 阶段1：代码持续预训练

#### 为什么需要？

基础 LLM 在通用语料上训练，代码数据占比很少。模型知道"if"是英文单词"如果"，但不知道 `if (x > 0) { ... }` 是编程语法。

**代码持续预训练** = 用海量代码数据继续训练基础模型，让模型"浸泡"在代码中。

#### 训练数据长什么样？

**格式A：普通代码文件**（纯文本，和通用预训练一样预测下一个token）

```python
# file: quicksort.py
def quicksort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[len(arr) // 2]
    left = [x for x in arr if x < pivot]
    middle = [x for x in arr if x == pivot]
    right = [x for x in arr if x > pivot]
    return quicksort(left) + middle + quicksort(right)
```

模型看到 `def quicksort(arr):` → 预测下一个 token 是换行 + 缩进。看到 `if len(arr)` → 预测 ` <= 1:`。

**格式B：FIM (Fill-In-the-Middle)**

```text
<|fim_prefix|>def quicksort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[len(arr) // 2]
    <|fim_suffix|>
    return quicksort(left) + middle + quicksort(right)
<|fim_middle|>
```

模型预测 `<|fim_middle|>` 处的内容：
```python
left = [x for x in arr if x < pivot]
middle = [x for x in arr if x == pivot]
right = [x for x in arr if x > pivot]
```

**为什么 FIM 重要？** 真实编程经常需要在已有代码中间插入内容（补全函数体、修复bug），FIM 让模型学会这种能力。

**数据比例：**

| 数据类型 | 比例 |
|---------|------|
| GitHub 代码 | ~85% |
| 代码文档、README、Stack Overflow | ~10% |
| 通用语料（防遗忘） | ~5% |

#### 训练配置

```python
base_model = load_model("general-llm")  # 加载基础模型

trainer = Trainer(
    model=base_model,
    data=load_github_code(),
    lr=1e-4,              # 比预训练低，防遗忘
    steps=100_000,
    fim_rate=0.5,         # 50% 数据用 FIM 格式
)
trainer.train()
```

#### 训练后能力

- 能正确写出语法合法的代码
- 知道常见 API 的使用方式
- 理解代码结构（函数、类、循环、异常处理）
- **还不能**很好地"按指令生成代码"

### 3.2 阶段2：代码指令微调 (Code SFT)

#### 为什么需要？

持续预训练后的模型能"续写代码"，但用户想要的是"按要求写代码"。

**指令微调** = 用 `(用户请求, 代码解答)` 成对数据训练，让模型学会"听指令"。

#### 训练数据长什么样？

**格式**：JSON，每个样本有 `instruction` / `input` / `output` 三个字段。

```json
{
  "instruction": "写一个 Python 函数，实现二分查找。函数接收一个已排序的整数列表和一个目标值，返回目标值的索引。如果目标值不存在，返回 -1。",
  "input": "",
  "output": "def binary_search(arr, target):\n    left, right = 0, len(arr) - 1\n    while left <= right:\n        mid = (left + right) // 2\n        if arr[mid] == target:\n            return mid\n        elif arr[mid] < target:\n            left = mid + 1\n        else:\n            right = mid - 1\n    return -1"
}
```

**带 input 的示例**（修复 bug）：

```json
{
  "instruction": "以下代码有 bug，请修复它。",
  "input": "def factorial(n):\n    if n == 0:\n        return 0\n    return n * factorial(n - 1)",
  "output": "def factorial(n):\n    if n <= 1:\n        return 1\n    return n * factorial(n - 1)"
}
```

**训练方式**：模型看到 `instruction + input` → 学习生成 `output`。**loss 只计算 output 部分**，instruction 部分被 mask 掉（设为 -100）。

**数据来源：**
- Stack Overflow、LeetCode 的问题-解答对
- 用更强模型（GPT-4、Claude）生成的合成数据
- 开源数据集（CodeAlpaca、Evol-Instruct-Code）

#### 训练配置

```python
trainer = SFTTrainer(
    model=code_pretrained_model,
    data=load_code_instructions(),
    lr=2e-5,              # 更低，数据量小
    epochs=3,
    loss_mask="output_only",  # 关键：只算 output 的 loss
)
trainer.train()
```

#### 训练后能力

- 按自然语言描述生成对应代码
- 修复给出的代码中的 bug
- 解释代码的功能
- **但**生成的代码不一定能运行（可能有语法/逻辑错误）

### 3.3 阶段3：代码强化学习 (Code RL)

#### 为什么需要？

SFT 的损失函数是"预测下一个token的概率"，只关心"生成的代码看起来对不对"，不关心"代码能不能运行"。

**强化学习** = 用"代码能否运行、测试是否通过"作为奖励信号，训练模型生成真正能工作的代码。

#### 训练数据长什么样？

**格式**：问题描述 + 初始代码 + 测试用例 + 多步推理轨迹

```json
{
  "problem": "实现一个 LRU 缓存。要求 get 和 put 操作都是 O(1)。",
  "initial_code": "class LRUCache:\n    def __init__(self, capacity: int):\n        pass\n    def get(self, key: int) -> int:\n        pass\n    def put(self, key: int, value: int) -> None:\n        pass",
  "test_cases": [
    {
      "input": ["LRUCache", "put", "put", "get", "put", "get", "get"],
      "args": [[2], [1,1], [2,2], [1], [3,3], [2], [3]],
      "expected": [null, null, null, 1, null, -1, 3]
    }
  ],
  "reasoning_steps": [
    "需要 O(1) 的 get/put，应该用哈希表 + 双向链表",
    "哈希表存 key → 链表节点的映射",
    "链表按访问时间排序，最近访问的在头部",
    "get 时：找到节点，移到头部，返回值",
    "put 时：如果 key 存在，更新值并移到头部；如果不存在，新建节点放头部，超容量删尾部"
  ],
  "final_code": "class DLinkedNode: ..."
}
```

**训练方式**：模型生成 `reasoning_steps` + `final_code`，外部程序运行测试，通过的越多奖励越高。

#### 核心方法：PPO + 单元测试奖励

```python
def reward_function(generated_code, test_cases):
    score = 0
    if compile_success(generated_code):
        score += 0.3
    passed = run_tests(generated_code, test_cases)
    score += 0.7 * (passed / len(test_cases))
    return score

# PPO 训练循环
for batch in dataloader:
    codes = model.generate(batch.problems)
    rewards = [reward_function(c, t) for c, t in zip(codes, batch.tests)]
    model.update_ppo(codes, rewards)
```

#### 更先进的方法

- **RLAIF**：用更强模型评判代码质量，不用人工写奖励规则
- **PRM (过程奖励模型)**：不仅看最终代码是否通过，还评判每一步推理是否正确

#### 训练后能力

- 生成的代码通过率大幅提升（~30% → ~80%+）
- 学会"先思考再编码"（Chain-of-Thought）
- 能处理更复杂的编程任务

### 3.4 阶段4：长程任务适配

前面的训练让模型"会编程"，但"工作 8 小时"的编程项目需要的能力远不止"写一段代码"。

#### 长程任务需要什么能力？

| 能力 | 说明 | 示例 |
|------|------|------|
| **长上下文记忆** | 记住整个项目结构 | "我之前说要使用 Redis" |
| **工具使用** | 调用外部工具 | "运行测试，看是否通过" |
| **规划与分解** | 把大任务拆成小步骤 | "先写接口，再写实现" |
| **自我纠错** | 发现错误并修正 | "测试失败了，原因是..." |
| **多轮交互** | 与用户持续沟通 | "用户说要登录功能，我需要确认..." |

#### 3.4.1 长上下文扩展

**问题**：基础模型通常只有 4K~8K 上下文，大型项目文件可能有 100K+ token。

**解法1：位置编码扩展**

```python
def extend_rope(model, original_length=4096, target_length=131072):
    scale = target_length / original_length
    model.rope_theta *= scale ** (2 / model.head_dim)
    return model
```

**解法2：长文本继续训练**

```python
long_context_data = load_long_documents(min_length=32768)

trainer = Trainer(
    model=model,
    data=long_context_data,
    max_seq_length=131072,  # 训练时就用长序列
    lr=1e-5,
)
```

**训练数据长什么样？**

```text
# file: full_project_readme.md (约 50000 tokens)

# 大型电商系统架构设计

## 1. 系统概述
本文档描述一个支持千万级并发的电商系统架构...
[此处省略 3000 行]

## 7. 数据库设计
### 7.3 订单表索引优化
在 2024 年 Q2 的压测中发现...
[此处省略 5000 行]

## 12. 缓存策略
Redis Cluster 采用三主三从架构...
[此处省略 2000 行]

## 问题：根据第 7 章的数据库设计，第 12 章的缓存应该如何配合？
```

模型必须记住文档开头的信息才能回答末尾的问题。

#### 3.4.2 工具使用 (Tool Use)

**核心思想**：训练模型生成"工具调用指令"，由外部程序执行，再把结果返回给模型。

**训练数据长什么样？**

```json
{
  "messages": [
    {"role": "user", "content": "帮我看看这个项目的测试覆盖率"},
    {"role": "assistant", "content": "我先看一下项目结构。"},
    {"role": "assistant", "content": "<tool>\n{\"name\": \"read_file\", \"parameters\": {\"path\": \"./package.json\"}}\n</tool>"},
    {"role": "tool", "content": "{\"scripts\": {\"test\": \"jest --coverage\"}}"},
    {"role": "assistant", "content": "<tool>\n{\"name\": \"run_shell\", \"parameters\": {\"command\": \"npm test\"}}\n</tool>"},
    {"role": "tool", "content": "PASS src/utils.test.js\nCoverage: 67% statements, 45% branches..."},
    {"role": "assistant", "content": "当前测试覆盖率：语句 67%，分支 45%。建议提高分支覆盖率..."}
  ]
}
```

模型学习在 `<tool>...</tool>` 标签中生成正确的 JSON，以及什么时候该调用什么工具。

#### 3.4.3 Agent 架构

**ReAct 模式**：`观察 → 思考 → 行动 → 观察 → ... → 完成`

```python
def react_agent(task, model, tools, max_steps=100):
    memory = [f"任务: {task}"]
    for step in range(max_steps):
        observation = "\n".join(memory)
        thought = model.generate(f"基于以下观察，思考下一步：\n{observation}\n思考：")
        action = model.generate(f"基于思考：{thought}\n行动：")
        if "<finish>" in action:
            return action
        result = execute_action(action, tools)
        memory.append(f"行动: {action}")
        memory.append(f"结果: {result}")
    return "达到最大步数限制"
```

**训练数据长什么样？** 完整的多步任务轨迹，包含错误和修正：

- 一个任务可能包含 50+ 步，训练模型进行长程规划和自我纠错

#### 3.4.4 推理时计算扩展 (Test-time Compute)

**核心洞察**：训练时投入更多计算，不如推理时让模型"多想一会儿"。

| 方法 | 原理 | 效果 |
|------|------|------|
| **Chain-of-Thought** | 先输出思考过程，再写代码 | 逻辑更清晰 |
| **Self-Consistency** | 生成 10 个答案，选测试通过率最高的 | 质量更稳 |
| **MCTS** | 把代码生成建模为搜索问题 | 找到最优解 |

```python
# Self-Consistency 示例
answers = []
for _ in range(10):
    answer = model.generate(prompt, temperature=0.8)
    score = evaluate_code(answer)
    answers.append((answer, score))
best_answer = max(answers, key=lambda x: x[1])
```

## 四、各阶段训练语料对比

| 阶段 | 数据格式 | 模型任务 | 数据规模 |
|------|---------|---------|---------|
| **通用预训练** | 纯文本：`《红楼梦》...` | 预测下一个 token | 数万亿 token |
| **代码持续预训练** | 代码文件 + FIM：`<|fim_prefix|>...` | 续写代码 / 填空 | 数千亿 token |
| **代码 SFT** | JSON：`{instruction, input, output}` | 条件生成代码 | 百万级样本 |
| **代码 RL** | 问题 + 测试用例 + 推理步骤 | 生成能通过测试的代码 | 十万级 |
| **长上下文扩展** | 超长文档（32K~128K token） | 记住前文回答后文问题 | 百亿级 token |
| **工具使用** | 多轮对话 + `<tool>{...}</tool>` | 生成正确的工具调用 | 十万级 |
| **Agent 训练** | 多步轨迹：观察→思考→行动→结果 | 长程规划与自我纠错 | 万级 |

## 六、关键洞察

**1. 数据质量 > 数据数量**

代码 RL 的数据量只有 SFT 的 1/10，但效果提升更大。因为"单元测试通过"是比"token 概率"更强的信号。

**2. 训练成本 vs 推理成本**

- 训练一个 70B 代码模型：数百万美元
- 推理时多用 10 倍计算（CoT、Self-Consistency）：几乎零额外成本

对于长程任务，**推理时计算扩展**往往是性价比最高的提升方式。

**3. 从"生成代码"到"完成项目"的鸿沟**

| 能力层级 | 示例 | 所需训练 |
|---------|------|---------|
| 写一行代码 | `x = a + b` | 代码预训练 |
| 写一个函数 | 实现快速排序 | 代码 SFT |
| 写能运行的代码 | 通过所有测试 | 代码 RL |
| 完成一个文件 | 实现一个类 | 长上下文 + 工具使用 |
| 完成一个模块 | 实现用户认证系统 | Agent 架构 |
| 完成一个项目 | 开发完整 Web 应用 | 多 Agent 协作 |
