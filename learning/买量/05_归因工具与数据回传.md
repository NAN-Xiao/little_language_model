## 8. 归因工具设置（Adjust/AppsFlyer）

### 8.1 为什么需要归因工具

```
问题：用户可能看了 Meta 广告、点了 Google 广告、最后从 Apple Search 下载
       到底算哪个渠道的功劳？

归因工具的作用：
  1. 确定用户从哪个渠道来（Last Click 归因）
  2. 追踪用户全生命周期行为
  3. 防止作弊安装
  4. 提供统一的 Dashboard 查看所有渠道数据
```

### 8.2 Adjust 设置步骤

```
1. 注册 Adjust 账号
   adjust.com

2. 创建 App
   - 选择平台（iOS/Android）
   - 输入 App Name 和 Bundle ID

3. 获取 App Token
   - 格式：abc123xyz

4. 集成 SDK
   iOS:
   ```swift
   import Adjust
   
   let config = ADJConfig(appToken: "abc123xyz", environment: ADJEnvironmentProduction)
   Adjust.appDidLaunch(config)
   
   // 追踪事件
   Adjust.trackEvent(ADJEvent(eventToken: "g3m52n"))
   ```
   
   Android:
   ```kotlin
   import com.adjust.sdk.Adjust
   import com.adjust.sdk.AdjustConfig
   
   val config = AdjustConfig(this, "abc123xyz", AdjustConfig.ENVIRONMENT_PRODUCTION)
   Adjust.onCreate(config)
   
   // 追踪事件
   val event = AdjustEvent("g3m52n")
   Adjust.trackEvent(event)
   ```

5. 创建事件（Events）
   - 在 Adjust Dashboard → Events → Create Event
   - 创建：
     - registration（注册）
     - tutorial_complete（完成新手引导）
     - level_achieved（完成关卡）
     - purchase（购买）
   - 获取每个事件的 Event Token

6. 关联广告平台
   - Adjust → Partner Setup
   - 添加：
     - Facebook
     - Google Ads
     - TikTok
     - Apple Search Ads
   - 输入各平台的 Account ID 或 App ID

7. 设置回调（Callbacks）
   - 把 Adjust 事件回传给各广告平台
   - 这样平台才能优化投放

8. 测试
   - Adjust 提供 Test Console
   - 可以模拟安装和事件，验证数据是否正确回传
```

### 8.3 AppsFlyer 设置步骤

```
AppsFlyer 和 Adjust 类似，选择其一即可。

主要区别：
  - Adjust：欧洲公司，界面简洁，价格相对低
  - AppsFlyer：以色列公司，功能更全面，价格较高

设置流程类似：
  1. 注册 → 创建 App → 获取 Dev Key
  2. 集成 SDK
  3. 创建事件
  4. 关联广告平台
  5. 测试验证
```

### 8.4 高质量数据回传标准

数据回传质量直接决定算法学习效果。低质量回传会让算法"学错"，越优化越偏。

**高质量回传的 4 个标准：**

| 标准 | 说明 | 具体要求 |
|------|------|---------|
| **准确** | 不漏传、不多传 | 每个真实事件都回传，虚假事件不回传 |
| **及时** | 尽快回传 | 事件发生后 24 小时内回传 |
| **完整** | 带完整参数 | 不仅传事件名，还要传金额、货币、商品 ID |
| **一致** | 口径统一 | 前后标准一致，不要今天改定义明天又改 |

**SLG 需要回传的关键事件：**

```
基础事件（必须）：
  - App Install（安装）→ 平台自动追踪
  - App Open（打开）→ 判断留存
  - Registration（注册）→ 判断注册率

付费事件（关键！）：
  - Purchase（购买）
    - 金额（value）
    - 货币（currency）
    - 商品 ID（content_id）

深度事件（优化用）：
  - Tutorial Complete（新手引导完成）
  - First Recharge（首充）
  - Alliance Joined（加入联盟）
  - VIP Level Up（VIP 升级）
```

**回传质量对比：**

```
❌ 低质量回传：
  {
    "event": "purchase",
    "timestamp": "2024-01-15"
  }
  → 只传了"购买"两个字，算法什么都学不到

✅ 高质量回传：
  {
    "event": "purchase",
    "value": 99.99,
    "currency": "USD",
    "content_id": "diamond_pack_5000",
    "content_type": "game_currency",
    "timestamp": "2024-01-15T08:30:00Z",
    "user_id": "user_12345",
    "platform": "iOS"
  }
  → 算法知道：这人用 iOS，花了 $99.99 买 5000 钻石
```

**常见回传错误：**

| 错误 | 例子 | 后果 |
|------|------|------|
| 漏传 | SDK 接好了但事件没触发 | 算法看不到付费，一直在找不付费的人 |
| 重复传 | 每次打开游戏都传 Purchase | 算法以为用户反复付费 |
| 金额错误 | 传人民币但标成 USD | 算法以为中国人人均 $648 |
| 事件名不统一 | 今天"purchase"明天"pay" | 算法识别不出来是同一个事件 |
| 测试数据没过滤 | 内部测试账号付费也回传 | 算法学了假数据 |

**检查回传质量的方法：**

```
Meta：
  - Events Manager 看 Purchase 事件的接收率和匹配率
  - 匹配率 < 90% → 有漏传

Google：
  - Firebase Console → Analytics → Events
  - 对比 in_app_purchase 事件数 vs 实际订单数

Adjust/AppsFlyer：
  - Dashboard 看 Raw Data
  - 差异 > 10% → 有问题
```

**技术实现要点：**

```
iOS 端（ATT 框架）：
  - iOS 14.5+ 需要用户授权追踪
  - 授权率影响数据完整性
  - 授权率 < 30% → 算法学习困难

服务器端回传（推荐）：
  - 客户端回传容易被拦截/篡改
  - S2S（Server-to-Server）回传更可靠
  - 先在自己的后台记录，再传给 Adjust/Meta
```
