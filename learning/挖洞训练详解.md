## 一、核心问题：挖洞训练是什么

普通预训练让模型"看到左边，预测右边"：

```
输入:  "def add(a, b):"
目标:  " return a + b"
```

但真实编程场景中，光标往往在代码**中间**，前后都有内容：

```python
def calculate(x):
    # 光标在这里 ↑
    return result
```

**挖洞训练** = 在训练数据中把代码的某段"挖掉"，让模型根据剩余内容填补。模型因此学会"根据上下文补全中间"的能力。

---

## 二、两种挖洞方式对比

| 维度 | FIM (Fill-In-the-Middle) | Span Corruption |
|------|-------------------------|-----------------|
| **挖洞位置** | 代码的"中间一段" | 随机任意位置 |
| **挖洞数量** | 1 个 | 多个 |
| **输入格式** | 特殊 token 标记边界 | 特殊 mask token |
| **适用场景** | IDE 代码补全、中间插入 | 通用预训练、局部修复 |
| **模型结构** | 不改，仍是 causal decoder | 不改，仍是 causal decoder |
| **实现层级** | 数据预处理 | 数据预处理 |

**共同点**：都不改模型结构，只改输入数据的构造方式。

## 三、FIM 的实现

### 3.1 数据分割

把完整代码切成三段：

```python
code = """def quicksort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[len(arr) // 2]
    left = [x for x in arr if x < pivot]
    middle = [x for x in arr if x == pivot]
    right = [x for x in arr if x > pivot]
    return quicksort(left) + middle + quicksort(right)"""

# 随机选一个分割点（按行）
prefix = """def quicksort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[len(arr) // 2]"""         # ← 前面部分

middle = """    left = [x for x in arr if x < pivot]
    middle = [x for x in arr if x == pivot]
    right = [x for x in arr if x > pivot]"""  # ← 被挖掉的部分

suffix = """    return quicksort(left) + middle + quicksort(right)"""  # ← 后面部分
```

### 3.2 拼接成训练样本

用**特殊 token** 标记三段边界：

```python
FIM_PREFIX = "<|fim_prefix|>"   # 标记 prefix 开始
FIM_SUFFIX = "<|fim_suffix|>"   # 标记 suffix 开始
FIM_MIDDLE = "<|fim_middle|>"   # 标记 middle 开始（从这里预测）

# 模型看到的完整输入
input_text = FIM_PREFIX + prefix + FIM_SUFFIX + suffix + FIM_MIDDLE + middle

# 实际内容：
"""<|fim_prefix|>def quicksort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[len(arr) // 2]<|fim_suffix|>    return quicksort(left) + middle + quicksort(right)<|fim_middle|>    left = [x for x in arr if x < pivot]
    middle = [x for x in arr if x == pivot]
    right = [x for x in arr if x > pivot]"""
```

**关键**：suffix（后面部分）被**提前**放到了输入的前面，在 `<|fim_middle|>` 的左边。

### 3.3 Loss Mask：只让模型学"填洞"

模型输入被 tokenize 后，构造 labels：

```
token_ids:  [101, 10, 20, 30, 102, 40, 50, 103, 60, 70, 80]
            ↑ prefix内容  ↑ suffix内容  ↑ FIM_MIDDLE ↑ middle内容

labels:     [-100, -100, -100, -100, -100, -100, -100, -100, 60, 70, 80]
             ↑ 这些位置设为 -100，不计算loss
                                         ↑ 只有这些位置计算loss
```

PyTorch 的 `CrossEntropyLoss(ignore_index=-100)` 会自动跳过 `-100` 的位置。

### 3.4 为什么模型能"看到两边"

模型 attention 仍然是 **causal**（只能看左边），但：

```
位置 0-6:  prefix 内容  ← 模型能看到
位置 7-8:  suffix 内容  ← 模型也能看到（在左边！）
位置 9:    FIM_MIDDLE   ← 标记token
位置 10-12: middle 内容 ← 模型要预测这些
```

因为 suffix 被**搬到了输入前面**，模型虽然只能"向左看"，但左边已经包含了 suffix。这是一种**数据层面的技巧**，不是模型结构的改动。

## 四、Span Corruption 的实现

### 4.1 随机挖多个洞

在代码中随机选择多个连续片段挖掉：

```python
# 原始代码
code = "def calculate(x, y):\n    result = x + y\n    return result"

# 随机挖掉两个 span
code_with_masks = "def <MASK>(x, y):\n    <MASK> = x + y\n    return result"
#          ↑ 挖掉 "calculate"          ↑ 挖掉 "result"
```

### 4.2 构造输入和目标

```python
# 模型输入（带 mask token）
input_tokens:  [101, 10, 103, 11, 12, 13, 103, 14, 15, 16]
               ↑"def" ↑MASK ↑"(x,y)" ↑MASK ↑"=x+y"

# 目标（按顺序列出被挖掉的内容）
target_tokens: ["calculate", "result"]
               ↑ 第一个 span   ↑ 第二个 span
```

### 4.3 和 FIM 的关键区别

| | FIM | Span Corruption |
|--|-----|-----------------|
| 挖洞位置 | 必须在代码"中间" | 任意位置 |
| 洞的数量 | 1 个 | 多个 |
| 目标组织 | middle 直接跟在 FIM_MIDDLE 后面 | 所有 mask 位置的目标按顺序排列 |
| 应用场景 | 光标在中间，补全后续 | 局部修复、填空 |

---

## 五、为什么不用 Attention Mask

### 5.1 不是 BERT 的那种 Mask

BERT 的做法（**我们不这样做**）：

```python
# BERT 的 attention mask
# [mask] 位置可以和所有位置互相看到（双向）
attention_mask = [
    [1, 1, 1, 1],   # 所有 token 双向可见
    [1, 1, 1, 1],
    ...
]
```

FIM / Span Corruption 的做法（**我们这样做**）：

```python
# Causal attention mask（三角矩阵，只向左看）
attention_mask = [
    [1, 0, 0, 0],   # 位置0 只能看自己
    [1, 1, 0, 0],   # 位置1 能看位置0,1
    [1, 1, 1, 0],   # 位置2 能看位置0,1,2
    [1, 1, 1, 1],   # 位置3 能看所有
]
# 完全不变！仍然是 causal decoder！
```

### 5.2 不改模型的三个原因

| 原因 | 说明 |
|------|------|
| **效率** | Causal attention 可以用 KV Cache 加速，双向不能 |
| **能力** | Causal decoder 既能生成又能理解，双向只能理解 |
| **简单** | 改数据比改模型简单100倍，零模型改动 |

## 六、混合训练策略

一个 batch 中混合多种训练方式：

```python
batch = [
    "def add(a, b):\n    return a + b",
    "class User:\n    def __init__(self, name):\n        self.name = name",
    "import numpy as np\narr = np.array([1, 2, 3])",
    ...
]

for code in batch:
    r = random.random()

    if r < 0.3:
        # 30% FIM（代码补全场景）
        sample = fim_transform(code)

    elif r < 0.6:
        # 30% Span Corruption（通用增强）
        sample = span_corruption_transform(code)

    else:
        # 40% 普通因果语言模型（保持续写能力）
        sample = causal_transform(code)
```

**为什么混合？**

- 只做 FIM → 模型只会"填中间"，不会"续写末尾"
- 只做 Span → 模型片段感太重，不会写完整函数
- 混合训练 → 同时保持"续写"和"补全"两种能力

## 七、关键代码实现

### 7.1 FIM 完整实现

```python
import random

def fim_transform(code_text, fim_rate=0.5):
    """把代码切成 prefix / middle / suffix"""
    if random.random() > fim_rate:
        return None, None, code_text  # 不做 FIM

    lines = code_text.split('\n')
    if len(lines) < 2:
        return None, None, code_text

    split_idx = random.randint(1, len(lines) - 1)

    prefix = '\n'.join(lines[:split_idx])
    middle = lines[split_idx]
    suffix = '\n'.join(lines[split_idx + 1:])

    return prefix, suffix, middle


def build_fim_sample(prefix, suffix, middle, tokenizer):
    """拼接成训练样本并构造 labels"""
    FIM_PREFIX = "<|fim_prefix|>"
    FIM_SUFFIX = "<|fim_suffix|>"
    FIM_MIDDLE = "<|fim_middle|>"

    # 拼接：prefix + suffix（上下文）+ FIM_MIDDLE（标记）+ middle（目标）
    text = f"{FIM_PREFIX}{prefix}{FIM_SUFFIX}{suffix}{FIM_MIDDLE}{middle}"
    tokens = tokenizer.encode(text)

    # 找到 FIM_MIDDLE 的位置
    middle_token = tokenizer.encode(FIM_MIDDLE)[0]
    middle_start = tokens.index(middle_token)

    # 构造 labels：FIM_MIDDLE 之前全设为 -100
    labels = [-100] * len(tokens)
    for i in range(middle_start + 1, len(tokens)):
        labels[i] = tokens[i]

    return tokens, labels
```

### 7.2 Span Corruption 完整实现

```python
def span_corruption_transform(tokens, mask_token_id,
                               mask_ratio=0.15, mean_span_length=3):
    """随机挖掉多个连续 token 片段"""
    n_tokens = len(tokens)
    n_to_mask = int(n_tokens * mask_ratio)

    if n_to_mask == 0:
        return tokens, tokens.copy()

    n_spans = max(1, n_to_mask // mean_span_length)
    span_starts = sorted(random.sample(range(n_tokens), n_spans))

    input_tokens = []
    target_tokens = []
    current_pos = 0

    for start in span_starts:
        span_len = random.randint(1, mean_span_length * 2)
        end = min(start + span_len, n_tokens)

        input_tokens.extend(tokens[current_pos:start])
        input_tokens.append(mask_token_id)  # 用 mask token 占位
        target_tokens.extend(tokens[start:end])

        current_pos = end

    input_tokens.extend(tokens[current_pos:])

    # labels：mask 位置填目标 token，其余 -100
    labels = [-100] * len(input_tokens)
    target_idx = 0
    for i, tok in enumerate(input_tokens):
        if tok == mask_token_id and target_idx < len(target_tokens):
            labels[i] = target_tokens[target_idx]
            target_idx += 1

    return input_tokens, labels
```

### 7.3 训练一步的完整流程

```python
import torch
import torch.nn.functional as F

def train_step(model, batch_codes, tokenizer, optimizer):
    batch_tokens = []
    batch_labels = []

    for code in batch_codes:
        r = random.random()

        if r < 0.3:
            # FIM
            prefix, suffix, middle = fim_transform(code)
            if prefix is not None:
                tokens, labels = build_fim_sample(prefix, suffix, middle, tokenizer)
            else:
                tokens = tokenizer.encode(code)
                labels = tokens.copy()

        elif r < 0.6:
            # Span Corruption
            tokens = tokenizer.encode(code)
            tokens, labels = span_corruption_transform(
                tokens,
                mask_token_id=tokenizer.encode("<|mask|>")[0]
            )

        else:
            # 普通因果 LM
            tokens = tokenizer.encode(code)
            labels = tokens.copy()

        batch_tokens.append(tokens)
        batch_labels.append(labels)

    # Padding
    max_len = max(len(t) for t in batch_tokens)
    padded_tokens = [t + [0] * (max_len - len(t)) for t in batch_tokens]
    padded_labels = [l + [-100] * (max_len - len(l)) for l in batch_labels]

    input_ids = torch.tensor(padded_tokens)
    labels = torch.tensor(padded_labels)

    # 前向 + loss
    logits = model(input_ids)
    loss = F.cross_entropy(
        logits.view(-1, logits.size(-1)),
        labels.view(-1),
        ignore_index=-100
    )

    loss.backward()
    optimizer.step()
    optimizer.zero_grad()

    return loss.item()
```

## 八、常见误区

### 误区1："FIM 让模型能双向看到上下文"

**错误理解**：模型 attention 变成了双向。

**正确理解**：attention 仍然是 causal（只能看左边），但 suffix 被**搬到了输入前面**（左边），所以模型"向左看"时就能看到 suffix。这是**数据技巧**，不是模型改动。

### 误区2："-100 是 attention mask"

**错误理解**：`-100` 阻止模型看到某些 token。

**正确理解**：`-100` 是 `CrossEntropyLoss` 的 `ignore_index`，只是**不计算 loss**，不影响 attention。模型仍然能看到这些 token（否则它就没有上下文了）。

### 误区3："挖洞训练必须改模型结构"

**错误理解**：需要加 encoder 或改 attention 机制。

**正确理解**：**零模型改动**。只需要在数据预处理阶段拼接文本、构造 labels。任何 causal decoder 都可以直接做挖洞训练。

### 误区4："FIM 和 Span Corruption 只能二选一"

**错误理解**：选了 FIM 就不能做 Span，反之亦然。

**正确理解**：可以**混合使用**。推荐比例：30% FIM + 30% Span + 40% 普通因果 LM，同时保持多种能力。

## 九、挖洞训练的扩展与变体

除了标准的 FIM 和 Span Corruption，代码预训练中还有许多衍生的"挖洞"方式。

---

### 9.1 FIM 的排列变体（PSM / SPM / SIP）

FIM 可以有三种 token 排列顺序，随机切换可以让模型更鲁棒：

| 变体 | 排列顺序 | 示例 |
|------|---------|------|
| **PSM** | Prefix → Suffix → Middle | `<|fim_prefix|>def f():<|fim_suffix|>return x<|fim_middle|>y = 1` |
| **SPM** | Suffix → Prefix → Middle | `<|fim_suffix|>return x<|fim_prefix|>def f():<|fim_middle|>y = 1` |
| **SIP** | Suffix → Middle → Prefix | `<|fim_suffix|>return x<|fim_middle|>y = 1<|fim_prefix|>def f():` |

**为什么需要多种排列？**

不同 IDE 和代码补全场景的触发位置不同：
- PSM：光标在中间，知道前后文 → 填中间
- SPM：光标在文件末尾，先看到"未来"的代码 → 反向生成
- SIP：专门用于"从后往前"的代码重构场景

```python
# 随机选择一种排列
order = random.choice(['PSM', 'SPM', 'SIP'])

if order == 'PSM':
    text = f"{FIM_PREFIX}{prefix}{FIM_SUFFIX}{suffix}{FIM_MIDDLE}{middle}"
elif order == 'SPM':
    text = f"{FIM_SUFFIX}{suffix}{FIM_PREFIX}{prefix}{FIM_MIDDLE}{middle}"
else:  # SIP
    text = f"{FIM_SUFFIX}{suffix}{FIM_MIDDLE}{middle}{FIM_PREFIX}{prefix}"
```

### 9.2 Diff-based Training（基于代码变更）

从 Git 提交历史中提取 `(修改前, 修改后)` 对，训练模型理解代码变更。

**数据格式**：

```
<|diff_before|>
def calculate(x):
    return x * 2
<|diff_after|>
def calculate(x):
    return x * 2 + 1
<|diff_message|>
修复了计算逻辑的边界错误
```

**两种训练目标**：

1. **给定修改前 + 描述，预测修改后**（代码修复）
2. **给定修改前 + 修改后，预测描述**（生成 commit message）

```python
def diff_transform(before_code, after_code, message=""):
    """构造 diff 训练样本"""
    text = f"<|diff_before|>{before_code}<|diff_after|>{after_code}"
    if message:
        text += f"<|diff_message|>{message}"
    return text
```

**优势**：
- 天然包含"意图 → 代码变更"的映射
- 不需要人工标注，直接从 GitHub 提取
- 训练模型理解"为什么这样改"而不仅是"改成什么"

### 9.3 Multi-Hole Infilling（多洞填充）

一次挖多个不连续的洞，洞之间有逻辑依赖。

**示例**：

```python
# 原始代码
def process(data):
    result = []
    for item in data:
        if item > 0:
            result.append(item * 2)
    return result

# 多洞填充
def process(data):
    <HOLE_1> = []
    for <HOLE_2> in data:
        if <HOLE_3> > 0:
            <HOLE_1>.append(<HOLE_3> * 2)
    return <HOLE_1>

# 模型需要同时推断：
# HOLE_1 = "result"（变量名一致性）
# HOLE_2 = "item"（循环变量）
# HOLE_3 = "item"（同一变量重复使用）
```

**关键挑战**：洞之间有依赖关系（HOLE_1 在三处出现，必须是同一个变量名）。

**实现方式**：

```python
def multi_hole_transform(code, n_holes=3):
    """随机选择 n 个 identifier 替换成 <HOLE_i>"""
    import re

    # 提取所有变量名/函数名
    identifiers = re.findall(r'\b[a-zA-Z_]\w*\b', code)
    # 选择出现多次且有逻辑关联的变量
    candidates = [name for name in set(identifiers)
                  if identifiers.count(name) >= 2]

    selected = random.sample(candidates, min(n_holes, len(candidates)))

    for i, name in enumerate(selected):
        code = code.replace(name, f"<HOLE_{i}>")

    return code, selected  # 返回修改后的代码和正确答案
```

### 9.4 Commit-based Training（基于 Git Commit）

从 Git 提交历史构造训练数据，形式为 `(上下文, Commit Message, 修改)`。

**数据格式**：

```json
{
  "context": "文件前面的代码...",
  "commit_message": "Add error handling for empty input",
  "before": "def parse(data):\n    return json.loads(data)",
  "after": "def parse(data):\n    if not data:\n        raise ValueError('Empty input')\n    return json.loads(data)"
}
```

**三种训练方式**：

| 方式 | 输入 | 目标 | 能力 |
|------|------|------|------|
| **方式A** | context + message + before | after | 根据描述修复代码 |
| **方式B** | context + before + after | message | 自动生成 commit message |
| **方式C** | context + message | before → after | 端到端代码修改 |

```python
# 方式A：根据描述修复代码
def commit_train_a(context, message, before, after):
    input_text = f"{context}\n# Task: {message}\n{before}"
    target = after[len(before):]  # 只预测新增的部分
    return input_text, target

# 方式B：生成 commit message
def commit_train_b(context, before, after):
    input_text = f"{context}\n{before}\n# Changed to:\n{after}"
    target = message
    return input_text, target
```

### 9.5 Type Inference Masking（类型推断训练）

专门挖掉类型注解，训练模型推断变量类型。

**示例**：

```python
# 原始代码
def process(data: List[int]) -> Dict[str, int]:
    result: Dict[str, int] = {}
    for item in data:
        result[str(item)] = item * 2
    return result

# 挖掉类型后
def process(data: <TYPE_1>) -> <TYPE_2>:
    result: <TYPE_2> = {}
    for item in data:
        result[str(item)] = item * 2
    return result

# 正确答案：
# TYPE_1 = List[int]
# TYPE_2 = Dict[str, int]
```

**为什么有效**：
- 类型信息是代码的"元数据"，理解类型是理解代码逻辑的关键
- 类型推断需要跨行理解（看到 `return result` 才知道 `result` 是什么类型）
- 对动态语言（Python）尤其重要

```python
def type_masking_transform(code):
    """挖掉 Python 类型注解"""
    import re

    # 匹配 : Type 模式
    type_pattern = r':\s*([A-Z][\w\[\],\s]*)'
    matches = re.findall(type_pattern, code)

    for i, match in enumerate(matches):
        code = code.replace(f': {match}', f': <TYPE_{i}>', 1)

    return code, matches
```

### 9.6 Identifier Masking（标识符掩码）

只挖掉变量名、函数名、类名，保留所有语法结构。

**示例**：

```python
# 原始代码
def calculate_average(numbers):
    total = sum(numbers)
    count = len(numbers)
    return total / count

# 标识符掩码后
def <ID_0>(<ID_1>):
    <ID_2> = sum(<ID_1>)
    <ID_3> = len(<ID_1>)
    return <ID_2> / <ID_3>

# 模型需要根据语义推断变量名：
# ID_0 = "calculate_average"（函数语义：计算平均值）
# ID_1 = "numbers"（参数：一组数字）
# ID_2 = "total"（总和）
# ID_3 = "count"（计数）
```

**与变量名替换的区别**：

| | 变量名替换 | 标识符掩码 |
|--|-----------|-----------|
| 操作 | 把 `total` 换成 `var_0` | 把 `total` 换成 `<ID_0>` |
| 训练目标 | 预测下一个 token（常规） | 专门预测被掩码的标识符 |
| 作用 | 防止过拟合特定变量名 | 强制理解语义推断命名 |

---

### 9.7 Permutation Language Modeling（排列语言模型）

XLNet 提出的方法：随机打乱 token 顺序，让模型预测被掩码的位置。

**在代码中的应用**：

```python
# 原始代码（token 顺序）
[def, process, (, data, ), :, return, data, ]

# 随机排列后
[data, def, process, return, (, ), :, data, ]
#  ↑ 1    ↑ 2   ↑ 3     ↑ 4    ↑ 5 ↑ 6 ↑ 7  ↑ 8

# 挖掉位置 3 和 7（原始顺序中的 "process" 和第二个 "data"）
[data, def, <MASK>, return, (, ), :, <MASK>, ]

# 模型需要根据排列后的上下文预测原始 token
```

**和 FIM 的区别**：
- FIM：保持 token 顺序，只挖连续片段
- Permutation：打乱顺序，可以挖任意位置

**缺点**：实现复杂，对代码这种结构化文本效果不如自然语言明显。

---

### 9.8 各种挖洞方式的适用场景

| 训练方式 | 最适合的场景 | 数据获取难度 |
|---------|------------|------------|
| FIM (PSM) | IDE 代码补全 | 容易（任意代码） |
| FIM (SPM/SIP) | 特殊编辑器场景 | 容易 |
| Span Corruption | 通用预训练增强 | 容易 |
| Diff-based | 代码修复、Code Review | 中等（需 Git 历史） |
| Multi-Hole | 变量命名、重构 | 中等 |
| Commit-based | Commit Message 生成 | 容易（Git 历史） |
| Type Masking | 类型推断、静态分析 | 容易 |
| Identifier Masking | 语义理解、命名建议 | 容易 |
| Permutation LM | 通用理解增强 | 困难 |

---

### 9.9 推荐组合

实际训练中，推荐按以下比例混合：

```python
training_mix = {
    "causal_lm": 0.30,           # 30% 普通因果语言模型（保持续写能力）
    "fim_psm": 0.20,             # 20% FIM PSM（标准补全）
    "fim_spm": 0.05,             # 5%  FIM SPM（反向生成）
    "span_corruption": 0.15,     # 15% Span Corruption（局部修复）
    "diff_based": 0.10,          # 10% Diff-based（代码变更）
    "type_masking": 0.10,        # 10% 类型推断
    "identifier_masking": 0.10,  # 10% 标识符掩码（语义理解）
}
```

**核心原则**：
- 保留至少 30% 的普通因果 LM，防止模型"忘记"怎么续写
- FIM 总量不超过 30%，避免过度偏向补全场景
- Diff/Type/Identifier 各占 10%，增强特定能力

## 总结

```
┌─────────────────────────────────────────┐
│           挖洞训练 = 数据预处理            │
│         （不改模型结构！）                │
├─────────────────────────────────────────┤
│                                         │
│  1. 数据层面：把代码切分/挖洞，拼接成      │
│     特殊格式（<|fim_prefix|>...）        │
│                                         │
│  2. Loss 层面：用 -100 mask 掉不需要      │
│     预测的位置，只让模型学"填洞"         │
│                                         │
│  3. 模型层面：完全不变，仍是 causal        │
│     decoder，attention 仍是三角矩阵     │
│                                         │
│  4. 扩展方式：PSM/SPM/SIP、Diff-based、   │
│     Multi-Hole、Type Masking 等          │
│                                         │
└─────────────────────────────────────────┘
```
