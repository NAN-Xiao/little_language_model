## 一、位置编码：让模型"知道词在第几个位置"

### 1.1 为什么需要位置编码？

Transformer 的 Attention 核心运算是 `Q @ K^T`，这个运算是**置换不变**的——你把句子里的词打乱顺序，Attention 计算出来的分数集合是一样的。模型根本分不清"猫抓老鼠"和"老鼠抓猫"的区别。

**位置编码就是给每个位置的词注入"序号信息"**。

### 1.2 演进路线：正弦 → RoPE → ALiBi

#### ① 正弦位置编码（Sinusoidal）— 原始 Transformer 方案

```
注入方式: x = token_embedding + PE[pos]

PE(pos, 2i)   = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))
```

**特点**：
- 在输入层加一次，后续层不再加
- 编码的是**绝对位置**："我在第 5 个位置"

**致命缺陷**：
- **深层稀释**：位置信息随着层数加深逐渐被变换冲淡
- **长度外推差**：训练时最大长度 512，推理时超过 512 就崩（因为没见过更长的位置编码）

**为什么被淘汰**：Llama、GPT-4 等现代模型都不用了。

---

#### ② RoPE（旋转位置编码）— 当前主流 ⭐

```
注入方式: 每层 Attention 内部，对 Q 和 K 做旋转变换
         q_rotated = rotate(q, θ_pos)
         k_rotated = rotate(k, θ_pos)

核心性质: 旋转后的 q 和 k 做点积，结果只取决于"角度差"
         → 编码的是相对位置："我离你 3 个位置"
```

**为什么比正弦强？**

| 对比项 | 正弦编码 | RoPE |
|--------|---------|------|
| 注入层级 | 输入层一次 | 每层都注入 |
| 编码内容 | 绝对位置 | 相对位置 |
| 深层稀释 | 会 | 不会 |
| 长度外推 | 差 | 较好（可配合 YaRN/NTK） |
| 计算方式 | 加法 | 旋转（保持向量长度） |

**关键理解——为什么旋转能编码相对位置？**

RoPE 把 d_k 维向量拆成 d_k/2 个 2D 向量对，每对用一个旋转矩阵：

```
旋转矩阵（角度 θ）:
  [cosθ  -sinθ]
  [sinθ   cosθ]

对向量 (x, y) 旋转:
  x' = x·cosθ - y·sinθ
  y' = x·sinθ + y·cosθ
```

**美妙性质**：
- 位置 m 的 q 旋转了 `m·θ`，位置 n 的 k 旋转了 `n·θ`
- 它们的点积结果只和 `(m-n)·θ` 有关
- → 位置差 = 5 时，无论 (m=10, n=5) 还是 (m=100, n=95)，点积都一样
- → **模型学到的是距离，不是绝对坐标**

**为什么 V 不旋转？**

Attention 公式: `softmax(Q·K^T) @ V`

位置信息通过 `Q·K^T` 的点积已经注入（角度差编码距离），V 只是被加权求和的内容载体，不需要位置信息。

---

#### ③ ALiBi — 长度外推之王

```
注入方式: 不给 embedding 加位置信息，
         而是在 Attention Score 上加一个和距离成比例的负偏置

score(i,j) = Q_i · K_j^T / √d_k - m × |i-j|
                          ↑
                    距离越远，扣分越多
```

**为什么外推强？**

训练和推理用同一套规则（距离越远惩罚越大），不像正弦/RoPE 依赖预计算的位置表。训练用 2K 长度，推理可以直接用到 100K+。

**为什么没成为绝对主流？**

短文本效果略逊于 RoPE，而且 RoPE 配合 YaRN/NTK 补丁后外推也不差。目前 RoPE 仍是生态最完善的选择。

---

### 1.3 三种编码的物理直觉

```
正弦编码: 给每个词发一张"座位号卡片"，入座后卡片就不管了
RoPE:     每次对话前（每层），根据两人的座位距离调整说话角度
ALiBi:    不记座位号，但默认"离得越远关系越弱"
```

## 二、注意力机制：从 MHA 到 MLA 的演进

### 2.1 统一框架：所有变体都基于 Scaled Dot-Product Attention

```
Attention(Q, K, V) = softmax(Q·K^T / √d_k) · V

直观理解:
  Q·K^T: "我（Query）和所有钥匙（Key）的匹配度"
  softmax: 匹配度归一化为概率（和为1）
  ·V:      按概率加权取内容（Value）
```

---

### 2.2 演进路线：MHA → GQA → MQA → MLA

**为什么要演化？核心矛盾：效果好 ↔ 推理快**

推理时的瓶颈不是计算，而是**KV Cache 显存**。每生成一个新 token，都要缓存前面所有 token 的 K 和 V。

---

#### ① MHA（Multi-Head Attention）— 标准多头

```
n_heads=12, d_model=768, d_k=64

Q: 12 个投影 → (B, 12, seq, 64)
K: 12 个投影 → (B, 12, seq, 64)
V: 12 个投影 → (B, 12, seq, 64)

每个头独立做注意力 → 12 个结果拼接 → 投影回 768 维
```

**KV Cache（每层）**: `2 × n_heads × seq × d_k` = 2 × 12 × seq × 64

**优点**：每个头有自己的 K/V，表达能力最强，不同头可以学到不同模式（有的关注语法，有的关注语义）

**缺点**：KV Cache 太大，长文本推理时显存爆炸。

---

#### ② GQA（Grouped-Query Attention）— 推荐平衡方案 ⭐

```
n_heads=12, n_kv_heads=3, d_k=64

Q: 12 个投影 → (B, 12, seq, 64)      ← 不变
K: 3 个投影  → (B, 3, seq, 64)       ← 变少了！
V: 3 个投影  → (B, 3, seq, 64)       ← 变少了！

关键操作: K/V 在 heads 维度重复 4 次
  K: (B, 3, seq, 64) → repeat → (B, 12, seq, 64)
  这样 Q 的 12 个头都能和 K 做运算
```

**KV Cache（每层）**: `2 × n_kv_heads × seq × d_k` = 2 × 3 × seq × 64 = **原来的 1/4**

**为什么效果没崩？**

- Q 投影还是独立的（12 个），每个头仍能学不同的"查询策略"
- K/V 被共享，相当于多个 Query 查同一个"知识库"
- LLaMA-2 论文：效果接近 MHA，推理快 30%

**为什么不是 MQA？**

MQA 所有头共享 1 个 K/V（n_kv_heads=1），Cache 最小但效果下降明显。GQA 在两者之间找到了甜点。

---

#### ③ MQA（Multi-Query Attention）— 极端压缩

```
n_heads=12, n_kv_heads=1

Q: 12 个投影
K: 1 个投影（所有 Q 共享）
V: 1 个投影（所有 Q 共享）
```

**KV Cache（每层）**: `2 × 1 × seq × d_k` = **原来的 1/12**

**代价**：所有头被迫共享同一个"记忆库"，表达能力受限。PaLM 用过，但后续模型大多转向 GQA。

---

#### ④ MLA（Multi-head Latent Attention）— DeepSeek 的创新

```
传统思路: 缓存完整的 K 和 V（形状大）
MLA 思路: 把 K/V 压缩成一个低维向量（latent vector），只缓存压缩后的向量

KV Cache 减少 90%+
```

**为什么这是革命性的？**

之前的路线（GQA/MQA）是"减少 K/V 的头数"，MLA 是"压缩 K/V 的内容"。DeepSeek-V2/V3 用 MLA + MoE，用 37B 激活参数达到 GPT-4 级别效果。

---

### 2.3 四种 Attention 的对比

| 变体 | n_kv_heads | KV Cache | 效果 | 代表模型 |
|------|-----------|----------|------|----------|
| MHA | = n_heads | 100% | 最好 | 原始 Transformer |
| GQA | n_heads/4 | 25% | 接近 MHA | LLaMA-2/3, Qwen |
| MQA | 1 | 8% | 明显下降 | PaLM, Falcon |
| MLA | 压缩向量 | ~10% | 接近 MHA | DeepSeek-V2/V3 |

---

### 2.4 注意力演进的核心逻辑

```
问题: 推理时 KV Cache 太大，长文本跑不动

解决方案的演化:
  MHA ──→ GQA ──→ MQA    （减少 K/V 头数，共享信息）
    ↓       ↓       ↓
  效果    平衡    速度
  最好    ⭐推荐   最快
    
  另一条路:
  MHA ──→ MLA              （压缩 K/V 内容本身）
    ↓
  效果保留 + 极致压缩
```

## 三、Normalization：训练稳定的"刹车系统"

### 3.1 为什么需要 Normalization？

深层网络（>6 层）训练时，梯度会随着反向传播逐层放大或缩小，导致：
- 梯度爆炸（loss 突然变成 NaN）
- 梯度消失（loss 完全不下降）

Normalization 把每层的输出缩放到合适的范围，保持数值稳定。

### 3.2 演进路线：Post-Norm → Pre-Norm → RMSNorm

---

#### ① Post-Norm — 原始 Transformer

```python
# 原始论文写法
x = x + Attention(x)
x = LayerNorm(x)

# 即: x = LayerNorm(x + Sublayer(x))
```

**问题**：残差连接 `x + Attention(x)` 之后，梯度要经过 LayerNorm 才能回传。LayerNorm 会压缩梯度，深层网络梯度越来越小 → **训练不稳定**。

**后果**：原始 Transformer 超过 12 层就很难训了。

---

#### ② Pre-Norm — 现代标配

```python
# 现在主流写法
x = x + Attention(LayerNorm(x))

# 残差连接是"高速公路"，梯度直通无阻
```

**为什么更稳定？**

```
Post-Norm 的梯度路径:
  loss → LayerNorm → Attention → ...
        ↑ 梯度被压缩

Pre-Norm 的梯度路径:
  loss ─────────────→ Attention → ...
    ↘────────────────→ x (残差直连)
        ↑ 有一条高速公路， gradient 不会消失
```

**代价**：理论上 Post-Norm 的表达能力更强（残差分支更复杂），但实际中 Pre-Norm 可以训更深的网络（100+ 层），总体效果更好。

---

#### ③ RMSNorm — Pre-Norm 的加速版

```python
# LayerNorm
x_norm = (x - mean(x)) / sqrt(var(x) + ε)

# RMSNorm（去掉 mean 中心化）
x_norm = x / sqrt(mean(x²) + ε)
```

**为什么去掉 mean 没问题？**

Transformer 里 LayerNorm 前面是 Attention 或 FFN，这些层的输出通常已经大致以 0 为中心了。去掉 mean 计算省了一步，速度更快。

**效果**：和 LayerNorm 持平或略好，计算更快。LLaMA、Qwen 都用它。

---

### 3.3 Normalization 演进的核心逻辑

```
问题: 深层网络梯度不稳定

Post-Norm: 在残差之后 Norm → 梯度被压 → 深层训不动
Pre-Norm:  在残差之前 Norm →  highway 保留 → 能训 100+ 层
RMSNorm:   Pre-Norm 的简化 → 更快，效果持平
```

## 四、FFN 激活函数：从 ReLU 到 SwiGLU

### 4.1 FFN 在模型中起什么作用？

Attention 做的是"信息筛选和聚合"（谁和谁相关），FFN 做的是"信息变换和存储"（具体知识的记忆）。

类比：Attention 是"开会讨论"，FFN 是"会后执行具体任务"。

### 4.2 演进路线：ReLU → GELU → SwiGLU

---

#### ① ReLU — 最早的选择

```python
FFN(x) = max(0, xW₁ + b₁)W₂ + b₂
```

**问题**：所有负值都变成 0，信息完全丢失。而且梯度在负区为 0，神经元"死亡"后就不再更新了。

---

#### ② GELU — 平滑版 ReLU

```python
GELU(x) = x × Φ(x)  # Φ 是标准正态分布的 CDF
```

**特点**：负值有小概率保留（平滑过渡），不是硬截断。BERT、GPT-2/3 用它。

---

#### ③ SwiGLU — 现在的主流 ⭐

```python
# SwiGLU 结构（注意有两套权重）
FFN(x) = (Swish(xW) ⊗ (xV))W₂

# 其中:
#   W, V: 两个并行的投影矩阵（门控分支和价值分支）
#   Swish: 平滑激活函数
#   ⊗: 逐元素乘法（门控机制）
#   W₂: 输出投影
```

**为什么效果好？**

- **门控机制**：`Swish(xW)` 决定哪些信息通过，`(xV)` 提供内容，两者相乘 = "有选择地传递信息"
- **双分支**：比单分支 ReLU/GELU 表达能力更强
- **Swish 激活**：自门控（self-gated），比 ReLU 更平滑

**代价**：多一个投影矩阵 V，参数量略增（但效果 consistently 更好）

---

### 4.3 三种激活的对比

| 激活 | 负值处理 | 平滑性 | 表达能力 | 代表模型 |
|------|---------|--------|---------|----------|
| ReLU | 硬截断（变0） | 不光滑 | 一般 | 原始 Transformer，你的项目 |
| GELU | 概率保留 | 光滑 | 较好 | BERT, GPT-3 |
| SwiGLU | 门控保留 | 光滑 | 最好 | LLaMA, Qwen, DeepSeek |

## 五、MoE（混合专家）：参数扩容的计算 trick

### 5.1 核心矛盾

想让模型更强 → 需要更多参数 → 推理更慢 → 用户等不起。

**MoE 的解决思路**：参数增加，但计算量不增加（每个 token 只激活部分参数）。

### 5.2 方案对比

```
Dense FFN（标准）:
  所有 token → 同一个 FFN → 输出
  参数量: d_model × d_ff × 2
  计算量: 和参数量成正比

MoE FFN（你的项目）:
  Router → 选 1 个专家 → 只过该专家
  总参数量: n_experts × d_model × d_ff × 2  （4倍）
  计算量: ≈ Dense（只激活 1 个专家）
```

### 5.3 负载均衡问题

**如果不加约束，Router 会学到"所有 token 都给专家 0"**，其他专家永远不被训练。

**解决方案**：负载均衡损失（Load Balance Loss）

```
LB = num_experts × Σ(f_e × P_e)

f_e = 实际分到专家 e 的 token 比例（不可导，纯统计）
P_e = Router 对专家 e 的平均概率（可导，影响梯度）

均匀分配时: f_e = P_e = 1/4 → LB = 4 × 4 × (1/4 × 1/4) = 1.0（最小值）
全部挤到专家 0: f_0 = P_0 ≈ 1.0 → LB = 4 × 1.0 = 4.0（很大）
```

**为什么 f_e 要 detach（不可导）？**

Router 只能通过概率 P_e 影响分配，不能直接控制 argmax 的硬决策。detach 确保梯度只流向 P_e，不通过分配结果回传。

## 六、训练流程：三阶段流水线

### 6.1 整体框架

```
预训练（Pre-training） → SFT → 对齐（RLHF/DPO）
   "学语言"              "学格式"   "学偏好"
```

---

### 6.2 预训练 — 学语言规律和世界知识

**数据**：海量互联网文本（万亿级 token）
**任务**：下一个词预测（Next Token Prediction）
**结果**：模型能续写、有常识、懂语法

**问题**：只会"续写"，不会"对话"。你问它"什么是 Transformer"，它会重复你的问题再回答。

---

### 6.3 SFT（Supervised Fine-Tuning）— 学对话格式

**数据**：人工编写的高质量 (问题, 答案) 对（万级）
**方法**：用问答对继续训练，但只让模型学习"答案"部分

**和预训练的区别**：

```python
# 预训练（text 模式）
输入:  [今天, 天气, 真, 好]
目标:  [天气, 真, 好, <eos>]   ← 所有位置都算 loss

# SFT（qa 模式，类似你的 --mode qa）
输入:  [问题: 什么是Transformer? 答案: 是一种神经网络架构]
目标:  [-100, -100, -100, ..., 是, 一, 种, ...]  ← 问题部分 mask 掉
```

**结果**：模型学会"你问我答"的格式，不再重复问题。

---

### 6.4 对齐 — 学"什么回答更好"

**SFT 的局限**：只能教"格式正确"，教不了"哪个回答更好"。

**例子**：
- 回答 A："投资要分散风险"（好）
- 回答 B："全部买妖股，保证翻倍"（坏，但格式正确）

**RLHF（三步走）**：

```
Step 1: 收集偏好数据
  同一个问题生成多个回答，人工排序

Step 2: 训练奖励模型（Reward Model）
  学会预测"人类会给这个回答打几分"

Step 3: PPO 强化学习
  模型生成 → 奖励模型打分 → 高分鼓励/低分抑制
```

**DPO（简化版 RLHF）**：

```
核心洞察: 可以直接从偏好数据推导出最优策略，
         不需要奖励模型和 PPO

RLHF: 模型 → 生成回答 → 奖励模型打分 → PPO 优化
DPO:  模型 → 直接比较"好回答"和"坏回答"的概率 → 优化

现在主流用 DPO（更简单、更稳定、效果往往更好）
```

## 七、训练指标：怎么看懂日志

### 7.1 Loss（损失）

```
loss = -log(模型给正确答案的概率)

loss=0.1  → 模型很有信心 (~90%) → 很好
loss=1.0  → 不太确定 (~37%) → 一般
loss=3.0  → 基本在猜 (~5%) → 很差
loss=4.16 → 完全随机（词表大小约 64 的均匀分布）
```

**越低越好！**

### 7.2 PPL（困惑度）

```
PPL = exp(loss)

物理意义: 模型每次选词，相当于从多少个"等概率选择"中猜

PPL=20  → 像从 20 个词里猜 → 不错
PPL=100 → 像从 100 个词里猜 → 一般
```

**比 loss 更直观**。

### 7.3 Train Loss vs Val Loss

| 情况 | Train Loss | Val Loss | 含义 | 对策 |
|------|-----------|----------|------|------|
| 正常 | ↓ | ↓ 且接近 Train | 模型在学习和泛化 | 继续训练 |
| 过拟合 | ↓↓ | ↑ 或持平 | 模型在"背答案" | 早停/加 dropout |
| 欠拟合 | 都很高 | 都很高 | 模型容量不够 | 增大模型/延长训练 |

---

## 八、学习率调度：三种策略

### 8.1 const（固定）

```
lr = 初始值（从头到尾不变）

适用: 微调、短训、已知最佳学习率
```

### 8.2 warmup_const（推荐）

```
step < warmup:  lr 从 0 线性增长到初始值
step >= warmup: lr = 初始值（恒定）

适用: 大多数场景，最稳定
类比: 开车先慢慢加速，然后匀速巡航
```

### 8.3 transformer（原论文）

```
lr = d_model^(-0.5) × min(step^(-0.5), step × warmup^(-1.5))

预热期:   lr ∝ step      （线性增长）
衰减期:   lr ∝ step^(-0.5)（平方根衰减）

适用: 大规模预训练，训练步数极多
```

## 附录 A：Conv2d 维度流转 —— 为什么 3 通道输入能变成 64 通道输出

> 本文记录 PyTorch `nn.Conv2d` 的核排列与维度变化，核心纠正一个常见误解：**输出通道数 ≠ 输入通道数的倍数**。

### A.1 一句话结论

```
Conv2d(in=3, out=64, kernel=3, stride=2)

核的 shape = (64, 3, 3, 3)
              ↑   ↑   ↑   ↑
           组数  每组核数 高   宽
           64    3    3   3

= 64 组，每组 3 个 3×3 的核，总共 192 个核
= 输出 64 个通道（每组产出 1 个通道）
```

### A.2 核的分组排列（关键！）

```
第 0 组（生成输出通道 0）:
  ├─ 核 [0,0,:,:] (3×3) ──→ 扫描输入通道 0（R）
  ├─ 核 [0,1,:,:] (3×3) ──→ 扫描输入通道 1（G）
  └─ 核 [0,2,:,:] (3×3) ──→ 扫描输入通道 2（B）
                              ↓ 3个结果逐像素相加
                        输出通道 0

第 1 组（生成输出通道 1）:
  ├─ 核 [1,0,:,:] (3×3) ──→ 扫描输入通道 0
  ├─ 核 [1,1,:,:] (3×3) ──→ 扫描输入通道 1
  └─ 核 [1,2,:,:] (3×3) ──→ 扫描输入通道 2
                              ↓ 3个结果逐像素相加
                        输出通道 1

...

第 63 组（生成输出通道 63）:
  ├─ 核 [63,0,:,:] (3×3) ──→ 扫描输入通道 0
  ├─ 核 [63,1,:,:] (3×3) ──→ 扫描输入通道 1
  └─ 核 [63,2,:,:] (3×3) ──→ 扫描输入通道 2
                              ↓ 3个结果逐像素相加
                        输出通道 63
```

### A.3 常见误解

| 误解 | 错误理解 | 正确理解 |
|------|---------|---------|
| ❌ 64÷3？ | "64 个核平分给 3 个通道" | ✅ 64 **组**，每组必须有 3 个核（覆盖所有输入通道） |
| ❌ 输出通道受限？ | "输入 3，输出只能是 3 的倍数" | ✅ 输出通道是独立超参数，和输入无关 |
| ❌ 1 对 1？ | "1 个输入通道对应 1 个输出通道" | ✅ 所有输入通道的信息被**混合相加**到每个输出通道 |

### A.4 为什么每组必须有 3 个核？

输出通道 0 的每个像素 = **输入 3 个通道在对应位置的卷积结果之和**。

如果只用 1 个核扫输入通道 0（R），那绿色通道和蓝色通道的信息就全丢了！所以**每组必须覆盖所有输入通道**。

### A.5 完整维度流转（以 1080p 图片为例）

```python
import torch
import torch.nn as nn

# 输入: 一张 3 通道的 1080×1920 图片
x = torch.randn(1, 3, 1080, 1920)  # (B, C, H, W)

# 定义卷积层
conv = nn.Conv2d(
    in_channels=3,      # 输入 3 通道
    out_channels=64,    # 输出 64 通道 ← 你定的超参数
    kernel_size=3,      # 3×3 小窗口
    stride=2,           # 每次跳 2 格 → 宽高减半
    padding=1           # 边缘补 1 圈 0
)

# 验证核的形状
print(conv.weight.shape)  # → torch.Size([64, 3, 3, 3])
#                         64组核，每组3个3×3

# 前向传播
out = conv(x)
print(out.shape)  # → torch.Size([1, 64, 540, 960])
#                       B   C    H    W
#                       1   64   540  960  (=1080/2, =1920/2)
```

### A.6 类比：64 个滤镜

把 Conv2d 想象成一台**同时装 64 个滤镜的相机**：

- 每个滤镜（输出通道）有 3 层玻璃片（对应 RGB 3 通道）
- 3 层玻璃片各自对原图的 R/G/B 做变形，然后光学叠加
- 64 个滤镜同时工作，产出 64 张风格不同的"特征照片"
- 这些"照片"不是人眼能看懂的图像，而是神经网络能理解的特征图

### A.7 从 3→64 到 256→4（VAE Encoder 的最后一步）

同样的原理，VAE Encoder 最后一层可能是：

```python
nn.Conv2d(in_channels=256, out_channels=4, kernel_size=3, stride=1, padding=1)
# 核 shape: (4, 256, 3, 3)
#          4组，每组256个核
#          256个中间特征图 → 压缩成 4 个 latent 通道
```

这里信息压缩得更狠：**256 个特征图被压成 4 个**。这 4 个通道每个都混合了大量视觉信息，人眼完全看不懂，但 Decoder 能还原出原图。

---

> **核心记忆锚点**：`out_channels` 不是算出来的，是你硬指定的核组数。输入通道数只决定**每组里有几个核**，不影响输出多少通道。
