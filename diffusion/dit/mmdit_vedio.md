# MMDiT Video: 从文字+图片输入到视频输出的完整流程

以真实数据为例：**1920×1080 横屏图片 + 文字提示 "一只橘猫在草地上从左走到右"** → 生成 4 秒 24fps 视频

---

```
用户输入                              用户得到
┌─────────────┐                     ┌──────────────┐
│ 图片 1920×1080 │                     │ .mp4 视频文件  │
│ + 文字提示    │                     │ 96帧 1088×1920│
└──────┬──────┘                     └──────▲───────┘
       │                                   │
       ▼                                   │
  ┌─────────┐                        ┌──────────┐
  │ T5-XXL  │                        │ 3D VAE   │
  │ 文本编码器│                        │ Decoder  │
  └────┬────┘                        └────▲─────┘
       │ text_emb                          │ z_0
       │ (1, 256, 4096)                    │ (1, 4, 24, 136, 240)

       └──────────┐     ┌──────────────────┘

            ┌──────────────────┐
            │  VideoMMDiT      │

            └──────────────────┘
```

## 2. 编码阶段：输入 → latent

### 2.1 文本编码（T5-XXL）

```
"一只橘猫在草地上从左走到右"
         │
         ▼  T5-XXL 编码器 (4.6B 参数，冻结不训练)
         │
  text_emb: (1, 256, 4096)
         │
         │  含义：256 个 token，每个 token 用 4096 维向量表示
         │  "橘猫"  → token 37, 2891 → 向量 [0.12, -0.03, 0.87, ...]
         │  "草地"  → token 102, 58  → 向量 [0.44, 0.91, -0.22, ...]
         │  "走"    → token 891      → 向量 [-0.55, 0.33, 0.11, ...]
         │  ...
```

### 2.2 图片/视频 → 3D VAE 编码

用户只给了一帧图片，模型需要先将其扩展为视频初始帧，其余帧用噪声填充或参考帧初始化。这里以编码一段 96 帧视频为例（训练时的流程）：

```
原始视频像素: (1, 3, 96, 1088, 1920)
              │   │   │    │     │
              │   │   │    │     └─ 宽度 1920
              │   │   │    └─────── 高度 1088 (1080 pad 到 1088，8 的倍数)
              │   │   └──────────── 帧数 96 (4秒 × 24fps)
              │   └──────────────── RGB 3通道
              └──────────────────── batch=1

         │
         ▼  3D VAE Encoder (4级下采样)

  ┌─────────────────────────────────────────────────────┐
  │  级1: Conv3d(3→128) + ResBlock×2 + 空间下采样2×     │
  │  (1, 128, 96, 544, 960)                             │
  │                                                     │
  │  级2: ResBlock×2 + 空间下采样2×                      │
  │  (1, 256, 96, 272, 480)                             │
  │                                                     │
  │  级3: ResBlock×2 + 空间2× + 时间2× 下采样            │
  │  (1, 256, 48, 136, 240)                             │
  │                                                     │
  │  级4: ResBlock×2 + 时间2× 下采样                     │
  │  (1, 512, 24, 136, 240)                             │
  │                                                     │
  │  投影: Conv3d(512→4)                                 │
  │  → μ: (1, 4, 24, 136, 240)                          │
  │  → log σ²: (1, 4, 24, 136, 240)                     │
  └─────────────────────────────────────────────────────┘

         │  重参数化: z_0 = μ + σ · ε,  ε ~ N(0, I)
         ▼

  latent z_0: (1, 4, 24, 136, 240)
              │  │   │    │    │
              │  │   │    │    └─ 宽 240 = 1920 / 8
              │  │   │    └────── 高 136 = 1088 / 8
              │  │   └─────────── 时间 24 = 96 / 4
              │  └─────────────── 4 通道 (压缩的特征)
              └────────────────── batch=1

  压缩比: 3×96×1088×1920 / 4×24×136×240 ≈ 597M / 3.1M ≈ 192×
```

## 3. Flow Matching：核心生成机制

### 3.1 Flow Matching 是什么？

Flow Matching 是一种**训练扩散模型**的方法，和 DDPM 是同级的概念：

```
┌──────────────────────────────────────────────────────────────┐
│  扩散模型训练方法对比:                                        │
│                                                              │
│  DDPM (2020):                                                │
│    加噪: x_0 → 逐步加噪 T 步 → x_T                          │
│    训练目标: 预测噪声 ε，loss = MSE(ε_θ, ε)                  │
│    推理: 逐步去噪，需要学噪声调度 β_t                         │
│                                                              │
│  Flow Matching (2023):  ← 本项目使用                         │
│    加噪: 直线路径 z_t = (1-t)·z_0 + t·z_1  一步到位         │
│    训练目标: 预测速度场 v，loss = MSE(v_θ, z_1 - z_0)        │
│    推理: 解 ODE 方程，不需要噪声调度                          │
│                                                              │
│  核心区别:                                                   │
│    DDPM: 离散马尔可夫链，每步只能从前一步到达                 │
│    FM:   连续 ODE 路径，任意 t 处都可以直接计算               │
│    FM 更简单: 不需要设计噪声调度，公式统一                    │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 数学原理：一条直线连接噪声和视频

```
  z_1 (纯噪声)                          z_0 (干净视频 latent)
    ●──────────────────────────────────────●
    t=1.0                                t=0.0

  中间任意时刻 t ∈ [0, 1]:

    z_t = (1 - t) × z_0  +  t × z_1

    例: t=0.0 → z_t = z_0            (干净视频)
        t=0.5 → z_t = 0.5·z_0+0.5·z_1 (一半信号一半噪声)
        t=1.0 → z_t = z_1            (纯噪声)

  速度场 (z_t 沿时间的变化率):

    v = d(z_t)/dt = z_1 - z_0       (直线路径的斜率，恒定不变!)

  训练目标: 让神经网络学会预测这个速度场

    v_θ(z_t, t, text) ≈ z_1 - z_0

    loss = MSE(v_θ(z_t, t, text),  z_1 - z_0)

  推理: 已知 z_1 (噪声), 求解 ODE 反向走到 z_0

    dz/dt = v_θ(z_t, t, text)

    从 t=1.0 向 t=0.0 积分 → 得到 z_0
```

### 3.3 训练过程：如何让模型学会预测速度

```
┌──────────────────────────────────────────────────────────────────────┐
│  训练一步的完整流程                                                   │
│                                                                      │
│  输入: z_0 (真实视频的 latent), text_emb (文本编码)                   │
│                                                                      │
│  ┌─ Step 1: 采样时间步 t ─────────────────────────────────────────┐ │
│  │  t ~ sigmoid(N(0,1))    ← logit-normal 分布                    │ │
│  │                                                                │ │
│  │  为什么不用均匀分布?                                            │ │
│  │    t≈0 (微量噪声): z_t≈z_0，预测太容易，浪费时间               │ │
│  │    t≈1 (纯噪声):   z_t≈z_1，信号太弱，梯度小                   │ │
│  │    t≈0.5 (中间):   半信号半噪声，最难预测，最有价值 ← 多采样  │ │
│  │                                                                │ │
│  │  logit-normal 分布峰值在 0.5，自动给中间区域更多采样            │ │
│  │  例: t 可能取到 0.42, 0.53, 0.38, 0.61, 0.47 ...              │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  ┌─ Step 2: 采样噪声 z_1 ────────────────────────────────────────┐ │
│  │  z_1 ~ N(0, I)                                                 │ │
│  │  形状与 z_0 相同: (B, 4, 24, 136, 240)                         │ │
│  │  每个元素独立从标准正态分布采样                                  │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  ┌─ Step 3: 直线插值构造 z_t ────────────────────────────────────┐ │
│  │  z_t = (1-t) × z_0 + t × z_1                                  │ │
│  │                                                                │ │
│  │  t 从 (B,) reshape 为 (B,1,1,1,1) 广播到 5D                   │ │
│  │  一步到位，不需要像 DDPM 那样逐步加噪                          │ │
│  │                                                                │ │
│  │  真实数据举例 (单个元素):                                       │ │
│  │    z_0[i] = 1.5  (干净值)                                      │ │
│  │    z_1[i] = 0.3  (噪声值)                                      │ │
│  │    t = 0.4                                                     │ │
│  │    z_t[i] = 0.6 × 1.5 + 0.4 × 0.3 = 1.02                    │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  ┌─ Step 4: CFG Dropout ──────────────────────────────────────────┐ │
│  │  以概率 p=10% 将 text_emb 替换为全零                           │ │
│  │                                                                │ │
│  │  为什么要丢弃文本?                                              │ │
│  │    推理时需要 CFG: 同时预测有文本和无文本的速度                  │ │
│  │    如果模型没见过"无文本"的情况，无条件预测会乱掉               │ │
│  │    所以训练时就要偶尔不给文本，让模型学会"无条件也能生成"        │ │
│  │                                                                │ │
│  │  每个样本独立判断 (不是整个 batch 一起丢):                      │ │
│  │    drop_mask = (rand(B) < 0.1)  → 约10%样本文本被清零          │ │
│  │    text_emb = text_emb.masked_fill(drop_mask, 0.0)             │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  ┌─ Step 5: 浮点 t → 整数索引 ────────────────────────────────────┐ │
│  │  t_int = round(t × 999)                                        │ │
│  │  VideoMMDiT 内部时间嵌入接收整数 [0, 999]                      │ │
│  │  例: t=0.0→0,  t=0.5→499,  t=1.0→999                          │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  ┌─ Step 6: VideoMMDiT 预测速度 ──────────────────────────────────┐ │
│  │  v_pred = VideoMMDiT(z_t, t_int, text_emb)                     │ │
│  │  输出形状: (B, 4, 24, 136, 240)                                │ │
│  │  每个位置的值表示"该位置从噪声走向视频的速度"                   │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  ┌─ Step 7: 计算损失 ────────────────────────────────────────────┐ │
│  │  v_target = z_1 - z_0       (直线路径斜率，常数)               │ │
│  │  loss = MSE(v_pred, v_target)                                   │ │
│  │                                                                │ │
│  │  真实数据举例:                                                  │ │
│  │    z_0[i] = 1.5,  z_1[i] = 0.3                                 │ │
│  │    v_target[i] = 0.3 - 1.5 = -1.2                              │ │
│  │    v_pred[i]  = -1.05  (模型预测，还没训好)                     │ │
│  │    loss_i = (-1.05 - (-1.2))² = 0.0225                         │ │
│  │    对所有元素求平均 → 标量 loss                                  │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  整个训练循环:                                                       │
│    for z_0, text_emb in dataloader:                                  │
│        loss = trainer.training_step(model, z_0, text_emb)            │
│        optimizer.zero_grad()                                         │
│        loss.backward()                                               │
│        clip_grad_norm_(model.parameters(), 1.0)                      │
│        optimizer.step()                                              │
└──────────────────────────────────────────────────────────────────────┘
```

### 3.4 推理过程：从噪声到视频的 ODE 求解

#### 3.4.1 Euler 法 (一阶精度，简单直接)

```
初始化:
  z = z_1 ~ N(0, I)        (1, 4, 24, 136, 240) ← 纯噪声
  null_emb = zeros_like(text_emb)                  ← 无条件嵌入
  dt = 1.0 / num_steps = 1/20 = 0.05

时间步序列:
  t = [1.00, 0.95, 0.90, 0.85, ..., 0.10, 0.05, 0.00]

每一步 (以第3步 t=0.90 为例):

┌───────────────────────────────────────────────────────────────────┐
│                                                                   │
│  当前状态: z = z_{0.95},  t = 0.90                               │
│                                                                   │
│  ① 有条件预测:                                                    │
│     t_tensor = full(B, 0.90)                                     │
│     t_int = round(0.90 × 999) = 899                              │
│     v_cond = VideoMMDiT(z, 899, text_emb)                        │
│     → "橘猫在草地上走"的方向                                      │
│                                                                   │
│  ② 无条件预测:                                                    │
│     v_uncond = VideoMMDiT(z, 899, null_emb)                      │
│     → "不知道要生成什么"的方向                                     │
│                                                                   │
│  ③ CFG 插值 (cfg_scale = 7.0):                                    │
│     v_cfg = v_uncond + 7.0 × (v_cond - v_uncond)                │
│                                                                   │
│     直觉解释:                                                      │
│       v_uncond = "合理视频"的方向 (基础运动学)                    │
│       v_cond   = "橘猫草地走"的方向 (语义引导)                   │
│       v_cond - v_uncond = "橘猫草地走"相对"普通视频"的偏移量     │
│       乘以 7.0 = 把这个偏移量放大7倍                              │
│       v_cfg = "普通视频" + 7倍"橘猫特征" = 极度贴合文本的视频    │
│                                                                   │
│     cfg_scale 效果:                                                │
│       1.0 → 等于有条件预测，无放大                                │
│       3.0 → 轻微引导，多样性好但可能不贴文本                      │
│       7.0 → 中等引导，质量和贴合度均衡 ← 常用默认                │
│       15+ → 强引导，严格贴文本但可能过饱和/不自然                 │
│                                                                   │
│  ④ Euler 步 (沿速度场反向走一步):                                  │
│     z = z - dt × v_cfg                                            │
│     z = z - 0.05 × v_cfg                                          │
│                                                                   │
│     为什么是减法?                                                  │
│       速度场 v 指向 t 增大方向 (z_0 → z_1)                       │
│       我们要从 t=1 走到 t=0，所以要反向: z_{t-dt} = z_t - dt·v   │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘

20步后的完整变化过程 (可视化):

  step  t      z 的状态
  ────────────────────────────────────────
   0    1.00   ░░░░░░░░░░  纯噪声，什么都看不见
   2    0.90   ░░░▒▒▒▒▒▒  微弱轮廓浮现
   4    0.80   ▒▒▒▒▒▓▓▓▓  大致色块可辨
   6    0.70   ▒▒▓▓▓▓▓▓▓  场景结构出现
   8    0.60   ▓▓▓▓▓███░  主体形状清晰
  10    0.50   ▓▓▓██████  主体和背景可分辨
  12    0.40   ████████░░  细节开始填充
  14    0.30   ████████▓▓  纹理逐渐丰富
  16    0.20   ██████████  高频细节
  18    0.10   ██████████  精细调整
  20    0.00   ██████████  完整 latent z_0

最后:
  z = z.clamp(-4.0, 4.0)     ← 夹断极端值，防止 VAE 解码失败
  return z                     ← 干净的视频 latent (1, 4, 24, 136, 240)
```

#### 3.4.2 Midpoint 法 (二阶精度，更少步数更高质量)

```
Euler 法的问题: 用起点速度走整步，走弯了
  z_t ──→ 用 v(z_t) 走 ──→ 偏离真实路径

Midpoint 法: 先走半步估计中点，再用中点速度走整步
  z_t ──→ 半步到 z_mid ──→ 用 v(z_mid) 走整步 ──→ 更贴近真实路径

每一步 (以 t=0.90 为例):

┌───────────────────────────────────────────────────────────────────┐
│                                                                   │
│  ┌─ 子步1: 在起点 z 估计速度 ───────────────────────────────────┐ │
│  │  v1_cond   = VideoMMDiT(z, t_int, text_emb)    ← 有条件    │ │
│  │  v1_uncond = VideoMMDiT(z, t_int, null_emb)     ← 无条件    │ │
│  │  v1 = v1_uncond + 7.0 × (v1_cond - v1_uncond)  ← CFG      │ │
│  │                                                              │ │
│  │  Euler 半步，走到中间点:                                      │ │
│  │  z_mid = z - (dt/2) × v1                                     │ │
│  │  t_mid = t - dt/2 = 0.90 - 0.025 = 0.875                    │ │
│  └──────────────────────────────────────────────────────────────┘ │
│                                                                   │
│  ┌─ 子步2: 在中间点 z_mid 估计速度 ────────────────────────────┐ │
│  │  v2_cond   = VideoMMDiT(z_mid, t_mid_int, text_emb)          │ │
│  │  v2_uncond = VideoMMDiT(z_mid, t_mid_int, null_emb)          │ │
│  │  v2 = v2_uncond + 7.0 × (v2_cond - v2_uncond)               │ │
│  │                                                              │ │
│  │  用中点速度走完整步:                                          │ │
│  │  z = z - dt × v2                                             │ │
│  └──────────────────────────────────────────────────────────────┘ │
│                                                                   │
│  代价: 每步调用 VideoMMDiT 4次 (Euler 只需 2次)                  │
│  收益: 15步 Midpoint ≈ 30步 Euler 的质量                         │
│        15步 × 4次 = 60次前向 vs 30步 × 2次 = 60次前向            │
│        同样计算量下质量更好                                       │
│                                                                   │
│  误差:                                                            │
│    Euler:    O(dt)   一阶误差，步长减半误差减半                  │
│    Midpoint: O(dt²)  二阶误差，步长减半误差减四分之一            │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

#### 3.4.3 两种方法对比总结

```
┌──────────────┬──────────────┬──────────────┬───────────────┐
│              │  Euler 法    │  Midpoint 法 │  说明         │
├──────────────┼──────────────┼──────────────┼───────────────┤
│ 精度         │ O(dt) 一阶   │ O(dt²) 二阶  │ Midpoint 更准 │
│ 每步调用次数 │ 2次 (有+无)  │ 4次          │ Midpoint 更贵 │
│ 推荐步数     │ 20~50        │ 10~20        │ Midpoint 更少 │
│ 总前向次数   │ 40~100       │ 40~80        │ 差不多        │
│ 同计算量质量 │ 较低         │ 较高         │ Midpoint 胜   │
│ 适用场景     │ 快速预览     │ 最终生成     │               │
└──────────────┴──────────────┴──────────────┴───────────────┘
```

### 3.5 CFG 深入：为什么需要两次前向传播

```
问题: 如果只用 v_cond = model(z_t, t, text_emb) 生成，会怎样？

  → 模型可能"偷懒"，不完全按文本生成
  → 因为训练数据中有各种视频，模型倾向于生成"平均"的视频
  → 比如 "橘猫在草地上走"，模型可能生成一只模糊的猫在绿色背景前

CFG 的解决: 让模型同时告诉我们两种情况

  v_uncond: "不看文本时，模型认为视频应该长什么样"
            → 基础运动学、自然视频统计规律
            → "应该有个东西在动，但不知道是什么"

  v_cond:   "看了文本后，模型认为视频应该长什么样"
            → 基础运动学 + 文本语义引导
            → "橘猫在草地上从左走到右"

  v_cond - v_uncond: 纯文本引导的方向
            → 去掉"通用视频"的基础，只保留"文本说了什么"
            → "橘猫"特征 + "草地"特征 + "从左到右"运动

  v_cfg = v_uncond + cfg_scale × (v_cond - v_uncond)
            → 保留基础运动学 + 放大文本特征
            → cfg_scale=7 意味着文本特征被强调7倍

直觉比喻:
  画师自己画 → v_uncond (画师凭感觉画，可能偏离要求)
  画师按需求画 → v_cond (画师看了需求，但可能不够贴合)
  CFG = "画师凭感觉" + 7×("按需求画" - "凭感觉画")
       = 基础功底 + 极度强调需求特征
```

## 4. VideoMMDiT 内部：核心神经网络

### 4.1 Embedding 层

```
z_t: (1, 4, 24, 136, 240)
         │
         ▼  VideoPatchEmbedding3D
         │  Conv3d(kernel=2×2×2, stride=2×2×2)
         │  把每 2×2×2 的 3D patch 投影为 d_model=1536 维向量
         │
         │  (1, 1536, 12, 68, 120)   ← 12个时间patch, 68×120个空间patch
         │       │
         │       ▼  flatten
         │
  vid_tokens: (1, 97920, 1536)
              │    │      │
              │    │      └─ d_model = 1536
              │    └──────── 总 patch 数 = 12 × 68 × 120 = 97,920
              └───────────── batch=1

         │  + 位置编码 (可学习)
         │  vid = vid_tokens + vid_pos_embed

t: (1,)
         │
         ▼  TimestepEmbedding
         │  正弦编码 → MLP(1536 → 6144 → 1536)
         │
  t_emb: (1, 1536)

text_emb: (1, 256, 4096)
         │
         ▼  Linear(4096 → 1536)
         │
  txt_tokens: (1, 256, 1536)
         │  + 位置编码
         │  txt = txt_tokens + txt_pos_embed
```

### 4.2 双流层 × 8 (VideoMMDiTBlock)

视频流和文本流各自有独立参数，通过时空联合注意力交互。

```
vid: (1, 97920, 1536)    txt: (1, 256, 1536)    t_emb: (1, 1536)
 │                          │                       │
 ▼                          ▼                       ▼
┌──────────────────────────────────────────────────────────┐
│              VideoMMDiTBlock × 8                         │
│                                                          │
│  ┌─ adaLN-Zero 条件调制 ──────────────────────────────┐ │
│  │  t_emb → vid_mod → [γ1, β1, α1, γ2, β2, α2]      │ │
│  │  t_emb → txt_mod → [γ1, β1, α1, γ2, β2, α2]      │ │
│  │                                                    │ │
│  │  vid_h = γ1_v × RMSNorm(vid) + β1_v              │ │
│  │  txt_h = γ1_t × RMSNorm(txt) + β1_t              │ │
│  └────────────────────────────────────────────────────┘ │
│                        │                                 │
│                        ▼                                 │
│  ┌─ STJointAttention ────────────────────────────────┐  │
│  │                                                    │  │
│  │  ══ Step 1: 空间联合注意力 ══                      │  │
│  │                                                    │  │
│  │  vid_h: (1, 97920, 1536)                          │  │
│  │    reshape → (12, 8160, 1536)   ← 12帧×8160空间   │  │
│  │    即每帧独立，有8160个空间patch                    │  │
│  │                                                    │  │
│  │  txt_h: (1, 256, 1536)                            │  │
│  │    expand → (12, 256, 1536)     ← 每帧看相同文本   │  │
│  │                                                    │  │
│  │  每帧做 Joint Attention:                           │  │
│  │    Q,K,V = [vid_frame ; txt] → 8160+256=8416 tokens│  │
│  │                                                    │  │
│  │    帧0: [■■■...8160个视频patch + ◆◆256个文本token] │  │
│  │           ↕ 双向注意力 ↕                            │  │
│  │    帧1: [■■■...8160个视频patch + ◆◆256个文本token] │  │
│  │    ...                                             │  │
│  │    帧11:[■■■...8160个视频patch + ◆◆256个文本token] │  │
│  │                                                    │  │
│  │    复杂度: O((8160+256)²) × 12 ≈ 8.5亿/帧         │  │
│  │                                                    │  │
│  │  ══ Step 2: 时间注意力 (纯视频, 无文本) ══         │  │
│  │                                                    │  │
│  │  vid: (1, 97920, 1536)                            │  │
│  │    reshape → (8160, 12, 1536)   ← 每个空间位置,   │  │
│  │    即同一空间位置跨12帧互看                         │  │
│  │                                                    │  │
│  │    位置(0,0): t0 ↔ t1 ↔ ... ↔ t11                │  │
│  │    位置(0,1): t0 ↔ t1 ↔ ... ↔ t11                │  │
│  │    ... (共8160个空间位置)                           │  │
│  │                                                    │  │
│  │    复杂度: O(12²) × 8160 ≈ 117万  ← 很小          │  │
│  │                                                    │  │
│  │  vid_final = 空间注意力输出 + 时间注意力输出        │  │
│  │  txt_final = 文本在12帧的输出取平均                │  │
│  └────────────────────────────────────────────────────┘ │
│                        │                                 │
│                        ▼                                 │
│  vid = vid + α1 × vid_attn     (残差)                    │
│  txt = txt + α1 × txt_attn     (残差)                    │
│                        │                                 │
│                        ▼                                 │
│  ┌─ 各自 FFN ────────────────────────────────────────┐  │
│  │  vid_h = γ2_v × RMSNorm(vid) + β2_v              │  │
│  │  vid = vid + α2 × FFN(vid_h)                      │  │
│  │  (1536 → 6144 → 1536, GELU激活)                   │  │
│  │                                                    │  │
│  │  txt_h = γ2_t × RMSNorm(txt) + β2_t              │  │
│  │  txt = txt + α2 × FFN(txt_h)                      │  │
│  └────────────────────────────────────────────────────┘ │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 4.3 合流 → 单流层 × 8 (VideoSingleStreamSTBlock)

```
vid: (1, 97920, 1536)    txt: (1, 256, 1536)
         │                          │
         └──────── concat ──────────┘
                    │
             x: (1, 98176, 1536)    ← 97920 + 256
                    │
                    ▼
┌──────────────────────────────────────────────────────┐
│        VideoSingleStreamSTBlock × 8                  │
│                                                      │
│  与双流层类似，但视频和文本共享参数：                  │
│                                                      │
│  1. 空间注意力: 每帧 [vid_frame ; txt] 做注意力      │
│     + 并行 FFN (FLUX 风格，一次线性算 QKV+FFN)       │
│                                                      │
│  2. 时间注意力: 只对视频，跨帧建模                    │
│                                                      │
│  单流层参数更少，但深度融合了视频和文本特征            │

                    │
             x: (1, 98176, 1536)
```

### 4.4 输出头

```

```

## 7. 时空联合注意力维度详解

以 1920×1080 / 4秒视频为例，**单次空间联合注意力**的数据流：

```
输入:
  vid: (1, 97920, 1536)     ← 12帧 × 8160空间位置
  txt: (1, 256, 1536)       ← 256个文本token

Step 1 — 空间 Joint Attention:
  vid → (12, 8160, 1536)    ← 每帧独立
  txt → (12, 256, 1536)     ← 文本复制到每帧

  每帧拼接: [vid_frame | txt] = (12, 8416, 1536)
                          ↑ 8160  +  256

  Q, K, V 线性投影:
    Q: (12, 24heads, 8416, 64)    ← 24个头，每头64维
    K: (12, 24heads, 8416, 64)
    V: (12, 24heads, 8416, 64)

  Attention 矩阵:
    (12, 24, 8416, 8416)          ← 每帧 8416×8416 的注意力矩阵
    视频patch可以看到文本token ✓
    文本token可以看到视频patch ✓
    双向交互！

  输出 split:
    vid_spatial_out: (12, 8160, 1536) → (1, 97920, 1536)
    txt_spatial_out: (12, 256, 1536)  → 帧间取平均 → (1, 256, 1536)

Step 2 — 时间 Attention (只有视频):
  vid → (8160, 12, 1536)      ← 每个空间位置，12帧互看

  Q, K, V:
    Q: (8160, 24, 12, 64)
    K: (8160, 24, 12, 64)
    V: (8160, 24, 12, 64)

  Attention: (8160, 24, 12, 12)  ← 只有12×12，极小

  输出: (1, 97920, 1536)
```

## 8. 完整推理流程总结

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  1. 用户输入                                                        │
│     "一只橘猫在草地上从左走到右" + 1920×1080 参考图                │
│                                                                     │
│  2. 编码                                                            │
│     文字 → T5-XXL → text_emb (1, 256, 4096)                       │
│     参考 → 3D VAE Encoder → z_0 (1, 4, 24, 136, 240)             │
│                                                                     │
│  3. Flow Matching 采样 (20步)                                       │
│     z_1 = N(0, I)           ← 纯噪声起点                          │
│     for step in 1..20:                                              │
│       v_cond   = VideoMMDiT(z_t, t, text_emb)   ← 有文本预测      │
│       v_uncond = VideoMMDiT(z_t, t, null_emb)    ← 无文本预测      │
│       v_cfg = v_uncond + 7.0 × (v_cond - v_uncond)  ← CFG引导    │
│       z_t = z_t - (1/20) × v_cfg               ← Euler步         │
│     z_0 = z_t                    ← 干净 latent                     │
│                                                                     │
│  4. VideoMMDiT 每次前向传播内部:                                    │
│     z_t → 3D PatchEmbed → 97,920 个 patch × 1536维                │
│     t   → TimestepEmbed → 1536维条件                               │
│     text → Linear → 256 token × 1536维                             │
│     ×8 双流层: 每帧空间+文本Joint Attn → 跨帧时间Attn → 各自FFN   │
│     ×8 单流层: 视频+文本合流 → 空间Attn+时间Attn+并行FFN          │
│     FinalLayer → unpatchify → v_θ (1, 4, 24, 136, 240)           │
│                                                                     │
│  5. 解码                                                            │
│     z_0 → 3D VAE Decoder → (1, 3, 96, 1088, 1920)                │
│     → 裁剪1088→1080 → 编码 → output.mp4                           │
│                                                                     │
│  总计算量: VideoMMDiT 被调用 20步 × 2次(CFG) = 40次前向传播        │
│  每次前向: 16层 Transformer，处理 ~98K token                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```
