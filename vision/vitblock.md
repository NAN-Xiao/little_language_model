以真实数据为例：**一张 224×224 的猫的 RGB 图片** → 输出 "猫" (1000 类中的第281类)

## 2. PatchEmbedding：图片 → patch 序列

### 2.1 为什么要切 patch？

```
NLP: 一句话 = 一串 token → 直接送进 Transformer
图片: 一张图 = 像素矩阵 → 像素太多 (224×224×3 = 150528), 注意力 O(n²) 算不动

解决: 把图片切成小块 (patch), 每个小块当一个 "token"
  就像把一句话拆成单词一样, 把一张图拆成小图块
```

### 2.2 切分过程

```
原图: (1, 3, 224, 224)
         │  │   │    │
         │  │   │    └─ 宽 224
         │  │   └────── 高 224
         │  └────────── RGB 3通道
         └───────────── batch=1

patch_size = 16, 所以把 224×224 的图切成 14×14 的网格:

  ┌────┬────┬────┬─···─┬────┐
  │ p1 │ p2 │ p3 │     │p14 │  ← 第1行, 14个patch
  ├────┼────┼────┼─···─┼────┤
  │p15 │p16 │    │     │p28 │  ← 第2行
  ├────┼────┼────┼─···─┼────┤
  │    │    │    │     │    │
  ├────┼────┼────┼─···─┼────┤
  │p183│    │    │     │p196│  ← 第14行
  └────┴────┴────┴─···─┴────┘

  每个 patch: 16×16 像素 × 3通道 = 768 个数
  总共: 14 × 14 = 196 个 patch
```

### 2.3 用 Conv2d 一步完成切分+投影

```
为什么不用循环一个一个切?
  循环太慢。Conv2d(kernel=16, stride=16) 一步搞定:
  - kernel=16: 每次看 16×16 的区域
  - stride=16: 每次跳 16 格 (不重叠)
  - 输出通道=768: 每个区域直接投影成 768 维

数学上等价于:
  手动切 patch → 展平成 768 维向量 → 线性投影到 768 维

维度变化:
  输入:  (1, 3, 224, 224)
  Conv2d(3→768, kernel=16, stride=16)
  输出:  (1, 768, 14, 14)
                ↑    ↑  ↑
                │    │  └─ 224/16 = 14 列
                │    └──── 224/16 = 14 行
                └───────── 768 维投影 (和每个patch展平后的768一一对应!)

  flatten(2): (1, 768, 14, 14) → (1, 768, 196)
    把空间维 14×14 合并成 196

  transpose(1,2): (1, 768, 196) → (1, 196, 768)
    交换通道和序列维, 变成 "196个token, 每个768维"
    这和 NLP 的 (batch, seq_len, d_model) 格式完全一致!

最终: (1, 196, 768) — 196 个 patch, 每个 768 维
```

## 3. [CLS] Token 和位置编码

### 3.1 [CLS] Token — 全局信息的收集器

```
问题: 196 个 patch, 每个都只代表局部, 谁来代表整张图?

  patch1: 只知道左上角有个尖尖的形状
  patch2: 只知道旁边有个圆圆的形状
  ...
  patch196: 只知道右下角是绿色的

  → 没有一个 patch 能回答 "这张图整体是什么"

解决: 造一个特殊的 token, 不对应任何 patch, 专门用来"收集"所有 patch 的信息
  这个特殊 token 叫 [CLS] — CLassification 的缩写

  初始: [CLS] 只是一个随机的 768 维向量, 什么都不代表
  经过 12 层注意力后:
    第1层: [CLS] 通过注意力 "看" 了所有 196 个 patch
           → [CLS] 开始聚合局部信息
    第6层: [CLS] 的 768 维里编码了更高级的特征
           → "尖尖的形状 + 圆圆的形状 = 可能是猫耳朵和猫眼睛"
    第12层: [CLS] = 全图的语义摘要
           → "这是一只猫"

维度变化:
  patch序列: (1, 196, 768)
  [CLS]:     (1,   1, 768)  ← 可学习参数, 初始随机
  concat:    (1, 197, 768)  ← 196个patch + 1个[CLS]

[CLS] 为什么能收集全局信息? 因为注意力的性质:

  [CLS] 做注意力时, q_CLS 会和所有 196 个 patch 的 k 计算分数:

    [CLS] 的 q: "我需要什么信息来理解整张图?"
    patch1 的 k: "我有左上角的边缘信息"
    patch50 的 k: "我有中间的纹理信息"
    ...

    注意力权重: [0.02, 0.05, ..., 0.15, ..., 0.01]
                  ↑ 不太重要    ↑ 这个对理解图很重要

    [CLS] 的输出 = 0.02×V₁ + 0.05×V₂ + ... + 0.15×V₅₀ + ...
                 = 按重要性加权汇总所有 patch 的内容
                 → 谁重要就看谁多一点 → 聚合了全图信息

  12 层重复这个过程 → [CLS] 越来越"懂"全图

类比: [CLS] 就像班长

  开学第一天: 班长啥都不知道 (随机初始化)
  和同学交流第1轮: 知道了每个人擅长什么 (第1层注意力)
  和同学交流第6轮: 知道了谁和谁是朋友, 小团体结构 (第6层)
  和同学交流第12轮: 对全班了如指掌 (第12层)

  最后老师问 "你们班整体怎么样?", 只问班长就够了
  不需要挨个问 196 个同学

为什么不用别的方法?

  方法1: 对 196 个 patch 取平均
    avg = (patch1 + patch2 + ... + patch196) / 196
    问题: 所有 patch 权重相同, 背景和主体一视同仁
    → 猫占图 10%, 草地占 90% → 平均后被草地淹没

  方法2: 用 [CLS] + 注意力 ← 本项目使用
    [CLS] 自动学会给重要 patch 高权重, 不重要的低权重
    → 猫的 patch 权重高, 草地权重低 → 准确提取关键信息

  方法3: 不用 [CLS], 直接在 196 个 patch 后面接分类头
    也可以! 有些 ViT 变体就是这么做的
    但 [CLS] 是原始论文的做法, 更主流
```

[CLS] 的训练与推理:

```
  [CLS] 是 nn.Parameter — 一个可学习的 768 维向量, 没有输入来源

  代码中:
    self.cls_token = nn.Parameter(torch.zeros(1, 1, d_model))
    初始全零, 训练中随梯度更新

  训练时: [CLS] 参数随梯度更新
    step 1:  [0.01, -0.02, 0.03, ...]   ← 刚开始学
    step 2:  [0.02, -0.01, 0.05, ...]   ← 梯度更新
    ...
    step N:  [1.23, -0.87, 0.15, ...]   ← 训练完, 固定下来

  推理时: [CLS] 参数不变, 每次直接拿来拼到序列头部
    图片A: [1.23, -0.87, 0.15, ...] + [patch1_A, patch2_A, ...] → "猫"
    图片B: [1.23, -0.87, 0.15, ...] + [patch1_B, patch2_B, ...] → "狗"
                    ↑ 同一个 [CLS] 向量, 参数不变

  初始值一样, 但经过 12 层注意力后输出不同
    → 因为每张图的 patch 不同, 注意力注入的信息不同
    → 同一个起点, 不同的终点

  类比: [CLS] 是一个"空杯子"
    杯子的形状 (参数) 训练好了就不变
    但每次往里倒的水 (注意力聚合的内容) 不同
    → 同一个杯子, 装了不同的水 → 得到不同的结果
```

[CLS] 在多模态模型中能看到文字吗?

```
  要分两个阶段:

  阶段1: ViT 内部 (拼接之前) — 没有文字, [CLS] 看不到文字

    [CLS] + 196个patch → 12层双向注意力 → [CLS] 聚合了全图视觉信息
    此时序列里根本没有文字, 所以看不到

  阶段2: Decoder 内部 (拼接之后) — 有文字, 但 [CLS] 也看不到文字!

    拼接后的序列: [CLS] p1 p2 ... p196 "这" "是" "什" "么" "?"
    位置:          0    1  2      196   197  198  199  200  201

    因果掩码第一行:
      [1 0 0 ... 0 0 0 0 0 0]   ← [CLS] 在位置0, 只能看到自己!

    所以 [CLS] 在 Decoder 里看不到任何文字 token

  这没关系, 因为 [CLS] 不需要看文字:

    [CLS] 的任务: 编码图像的全局视觉信息 (在 ViT 里已经完成了)
    文字 token 的任务: 参考 [CLS] 和所有视觉 token 来生成回答

    "这" (位置197): 能看到位置0~197 → [CLS] + 所有patch + 自己 ✓
    "是" (位置198): 能看到位置0~198 → 所有视觉 + "这" + 自己 ✓

    → 文字看得到 [CLS], [CLS] 不需要看文字
    → 信息流是单向的: 视觉 → 文字, 不需要反过来

  类比: 你拿着猫的照片问朋友"这是什么?"
    朋友看照片回答你 → 照片不需要"看"你的问题
    但你的朋友需要看照片才能回答
```

### 3.2 位置编码 — 告诉模型每个 patch 在哪

```
问题: Transformer 的注意力不知道位置 (patch1 和 patch100 看起来没区别)

解决: 给每个位置加一个可学习的向量
  位置0: [CLS]   → 加 pos_embed[0]
  位置1: patch_左上 → 加 pos_embed[1]
  位置2: patch_左上右 → 加 pos_embed[2]
  ...
  位置196: patch_右下 → 加 pos_embed[196]

维度变化:
  x:         (1, 197, 768)  ← patch序列+[CLS]
  pos_embed: (1, 197, 768)  ← 可学习, 训练中更新
  x + pos_embed: (1, 197, 768)
    逐元素相加, 每个位置的 768 维向量加上了位置信息

为什么用可学习位置编码, 不用正弦编码?V
  ViT 原论文实验: 可学习的比正弦的效果略好
  图像的 2D 空间关系比 1D 序列更复杂, 可学习的更灵活
```

## 4. ViTEncoderBlock — 双向自注意力 + FFN

### 4.1 和 Decoder-Only 的关键区别

```
┌──────────────────────────────────────────────────────────┐
│  Decoder (语言模型)          │  Encoder (ViT)            │
│  ──────────────────────      │  ──────────────────       │
│  因果掩码: 只看过去          │  无掩码: 所有patch互相看   │
│  mask = 下三角 (0/1)         │  mask = None (全1)        │
│  自回归: 逐个生成token       │  全局理解: 一次看完全图    │
│                                                              │
│  因果掩码示例 (4个token):     │  ViT注意力 (4个patch):     │
│  [1 0 0 0]  token0只看自己   │  [1 1 1 1]  p0看所有      │
│  [1 1 0 0]  token1看0,1      │  [1 1 1 1]  p1看所有      │
│  [1 1 1 0]  token2看0,1,2    │  [1 1 1 1]  p2看所有      │
│  [1 1 1 1]  token3看所有     │  [1 1 1 1]  p3看所有      │
│                                                              │
│  为什么? 生成时不能偷看未来   │  为什么? 分类需要全局理解  │
│  "猫吃" → 预测"鱼"          │  猫耳朵和猫尾巴要建立联系  │
└──────────────────────────────────────────────────────────┘
```

### 4.2 ViT 为什么不需要双流/联合注意力？

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  不同架构的注意力对比:                                           │
│                                                                  │
│  ┌─────────────┬──────────────┬──────────────┬──────────────┐   │
│  │             │ ViT          │ 语言模型      │ MMDiT Video  │   │
│  │ ─────────── │ ──────────── │ ──────────── │ ──────────── │   │
│  │ 输入类型    │ 只有图像     │ 只有文本     │ 图像 + 文本  │   │
│  │ 注意力方式  │ 单流自注意力 │ 单流自注意力 │ 双流→合流    │   │
│  │ 掩码        │ 无 (双向)    │ 因果掩码     │ 空间双向     │   │
│  │             │              │              │ 时间双向     │   │
│  │ 文本交互    │ 无           │ 不需要       │ 联合注意力   │   │
│  │ 位置编码    │ 可学习,1次   │ 正弦/RoPE    │ 可学习,1次   │   │
│  │ 复杂度      │ 最简单       │ 简单         │ 最复杂       │   │
│  └─────────────┴──────────────┴──────────────┴──────────────┘   │
│                                                                  │
│  ViT 不需要双流, 因为:                                          │
│                                                                  │
│  1. 只有一种输入 — 图像 patch, 没有文本流需要交互              │
│     MMDiT 需要双流是因为视频和文本是两种模态,                  │
│     各自统计特性不同, 早期需要独立参数各自提取特征              │
│     ViT 只有图像, 所有 patch 是同质的, 共享一套参数就行        │
│                                                                  │
│  2. 分类任务, 不需要文本引导                                    │
│     MMDiT 的文本流是为了引导生成方向 ("画一只猫")              │
│     ViT 只是分类 ("这是什么?"), 不需要文本条件                 │
│                                                                  │
│  3. 双向注意力已经足够                                          │
│     每个 patch 可以看到所有其他 patch                           │
│     12层双向注意力足以让 [CLS] 聚合全图信息                    │
│     不需要额外的跨模态交互机制                                  │
│                                                                  │
│  什么时候 ViT 需要更复杂的结构?                                 │
│    → 做多模态任务 (看图说话) 时, 就需要 VisionLanguageModel    │
│    → 此时 ViT 只是视觉编码器, 后面还要接 Decoder 处理文本      │
│    → 详见第 6 节                                                │
│                                                                  │
│  类比:                                                           │
│    ViT = 一个人安静地看完整张图, 然后回答 "这是什么"           │
│    语言模型 = 一个人逐字写文章, 写下一个字时只能看前面         │
│    MMDiT = 两个人(视频+文本)先各自研究, 再交流合作完成任务     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 4.2 单个 EncoderBlock 的完整流程

```
输入 x: (1, 197, 768)  ← [CLS] + 196个patch

  ┌──────────────────────────────────────────────────────┐
  │  子层1: 自注意力 (无掩码, 双向)                      │
  │                                                      │
  │  normed = LayerNorm(x)       → (1, 197, 768)        │
  │                                                      │
  │  MultiHeadAttention:                                  │
  │    q = k = v = normed (自注意力, 不分cross)          │
  │    mask = None (双向, 所有点互相看)                   │
  │                                                      │
  │    内部维度变化:                                      │
  │      w_q(normed): (1,197,768) → (1,197,768)         │
  │      .view: (1,197,768) → (1,197,12,64)              │
  │        768 = 12头 × 64维/头                           │
  │      .transpose: (1,12,197,64)                        │
  │        把头维提前, 每个头独立算注意力                 │
  │                                                      │
  │      scores = q @ k^T / √64                          │
  │        (1,12,197,64) @ (1,12,64,197)                 │
  │        → (1,12,197,197)                              │
  │        每个patch对197个位置的注意力分数              │
  │        (没有mask! 全部可见)                           │
  │                                                      │
  │      weights = softmax(scores)  → (1,12,197,197)     │
  │      out = weights @ v          → (1,12,197,64)      │
  │                                                      │
  │      拼回: .transpose + .view → (1,197,768)          │
  │      w_o: (1,197,768) → (1,197,768)                  │
  │                                                      │
  │  x = x + Dropout(attention_out)  ← 残差连接          │
  │  → (1, 197, 768)                                     │
  └──────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────┐
  │  子层2: FFN (逐位置非线性变换)                        │
  │                                                      │
  │  normed = LayerNorm(x)       → (1, 197, 768)        │
  │                                                      │
  │  Linear(768 → 3072): 升维4倍  → (1, 197, 3072)     │
  │    为什么升维? 单层线性只有仿射变换, 没非线性        │
  │    升维后经过 ReLU 才有非线性表达能力                │
  │                                                      │
  │  ReLU: 负数变0                → (1, 197, 3072)       │
  │  Dropout:                     → (1, 197, 3072)       │
  │  Linear(3072 → 768): 降维回来 → (1, 197, 768)       │
  │  Dropout:                     → (1, 197, 768)       │
  │                                                      │
  │  x = x + Dropout(ffn_out)     ← 残差连接             │
  │  → (1, 197, 768)                                     │
  └──────────────────────────────────────────────────────┘

输出: (1, 197, 768) — 形状和输入完全一样, 但每个token融合了全局信息

经过12层后:
  [CLS] 的768维向量 = 聚合了全图196个patch的信息
  patch_i 的768维向量 = 自身信息 + 和其他所有patch的关系
```

## 5. 分类头 — [CLS] → 类别

```
12层EncoderBlock输出: (1, 197, 768)
  LayerNorm:           (1, 197, 768)

  取 [CLS] 的输出: x[:, 0] → (1, 768)
    为什么只取[CLS]?
      [CLS] 经过12层注意力, 已经"看"过所有patch
      它的768维向量包含了全图的语义信息
      不需要再汇总196个patch, 一个[CLS]就够了

  分类头: Linear(768 → 1000) → (1, 1000)
    768维 → 1000维, 每维对应一个ImageNet类别
    输出的是 logits (未归一化分数)

  softmax: (1, 1000) → 概率分布
    例如: [..., 0.92, ...]  类别281(猫)概率最高
```

## 6. 多模态扩展: VisionLanguageModel (看图说话)

### 6.1 核心思路

```
问题: ViT 只能分类, 怎么让模型"看图说话"?

思路 (LLaVA 风格 — 当前最主流的多模态架构):
  1. ViT 编码图像 → 一组 "视觉token"
  2. Projector 把视觉token对齐到文本空间
  3. 视觉token 拼在文本token前面
  4. 一起送进 Decoder 做自回归生成

  就像给语言模型"看"了一组特殊的文字

  ┌──────────────────────────────────────────────────────────────┐
  │  为什么用 Concat 拼接, 不用 Cross-Attention?                 │
  │                                                              │
  │  Cross-Attention (早期, Flamingo/BLIP-2):                    │
  │    Decoder 里加额外的 cross-attn 层, 文本查询视觉特征        │
  │    优点: 视觉token不占文本序列                               │
  │    缺点: 要改LLM结构, 不能直接复用预训练权重                 │
  │                                                              │
  │  Concat 拼接 (主流, LLaVA/InternVL/Qwen2-VL):               │
  │    视觉token直接拼在文本前, 送进同一个Decoder               │
  │    优点: 不改LLM结构, 复用预训练权重, 实现简单              │
  │    缺点: 视觉token占序列长度, 注意力计算量稍大               │
  │                                                              │
  │  Concat 赢了的原因: 不改 LLM 结构 = 直接用 Llama/Qwen 权重  │
  └──────────────────────────────────────────────────────────────┘
```

### 6.2 函数原型

```python
class ImageTextProjector(nn.Module):
    """图像特征 → 文本空间 投影器"""
    def __init__(self, vit_d_model: int, llm_d_model: int)
    def forward(self, image_features: torch.Tensor) -> torch.Tensor
        # image_features: (B, N_img, vit_d_model) → (B, N_img, llm_d_model)

class VisionLanguageModel(nn.Module):
    """图文多模态模型: 输入图像+文字 → 输出文字"""
    def __init__(
        self,
        vit: VisionTransformer,       # 预训练的 ViT 图像编码器
        decoder: nn.Module,            # 文本 Decoder (来自 Transformer)
        output_proj: nn.Linear,        # Linear(d_model, vocab_size)
        projector: ImageTextProjector, # 图文投影层
        pad_token_id: int = 0,
        freeze_vit: bool = True,       # 冻结 ViT 参数
    )
    def forward(self, pixel_values, input_ids) -> torch.Tensor
        # pixel_values: (B, C, H, W) — 输入图像
        # input_ids:    (B, T)       — 输入文本
        # 返回: logits  (B, N_img + T, vocab_size)

    def generate(self, pixel_values, input_ids, max_new_tokens=128, ...) -> torch.Tensor
        # 自回归生成, 返回 (B, T + new_tokens)
```

### 6.3 "拼在前面" 到底怎么拼?

```
torch.cat 沿序列维度 (dim=1) 拼接:

  视觉 token: 197 个, 每个 768 维           形状: (1, 197, 768)
    位置0:   [CLS]   [0.12, -0.03, 0.87, ...]
    位置1:   patch1  [0.44, 0.91, -0.22, ...]
    位置2:   patch2  [-0.55, 0.33, 0.11, ...]
    ...
    位置196: patch196 [0.28, -0.67, 0.45, ...]

  文本 token: 5 个, 每个 768 维              形状: (1, 5, 768)
    位置0:   "这"    [0.15, 0.82, -0.31, ...]
    位置1:   "是"    [-0.09, 0.47, 0.63, ...]
    位置2:   "什"    [0.71, -0.18, 0.29, ...]
    位置3:   "么"    [0.33, 0.55, -0.44, ...]
    位置4:   "?"     [-0.22, 0.14, 0.76, ...]

  拼接后:                                    形状: (1, 202, 768)
    位置0:   [CLS]   [0.12, -0.03, 0.87, ...]  ← 视觉
    位置1:   patch1  [0.44, 0.91, -0.22, ...]  ← 视觉
    位置2:   patch2  [-0.55, 0.33, 0.11, ...]  ← 视觉
    ...
    位置196: patch196 [0.28, -0.67, 0.45, ...] ← 视觉
    ─────────── 拼接线 ───────────
    位置197: "这"    [0.15, 0.82, -0.31, ...]  ← 文本
    位置198: "是"    [-0.09, 0.47, 0.63, ...]  ← 文本
    位置199: "什"    [0.71, -0.18, 0.29, ...]  ← 文本
    位置200: "么"    [0.33, 0.55, -0.44, ...]  ← 文本
    位置201: "?"     [-0.22, 0.14, 0.76, ...]  ← 文本

  代码: combined = torch.cat([image_embeds, text_embeds], dim=1)
    dim=1 = 序列维度, 把 197 和 5 接在一起 → 202

  拼接后对模型来说:
    202 个 token 都是 768 维向量, 没有区别
    模型不知道哪些是图像、哪些是文字
    只是通过注意力权重自然学会: 前面的是图像信息, 后面的是文字信息

  为什么视觉 token 在前面, 不在后面?
    因为因果掩码: 文本 token 必须能看到视觉 token
    如果视觉在后面 → 文本在因果掩码下看不到视觉 → 无法看图说话
    视觉在前面 → 文本能看到前面所有视觉 token ✓
```

### 6.4 训练时完整维度流转 (模拟数据)

```
输入: 一张 224×224 猫图片 + 文字 "这是什么?"
      图像: (1, 3, 224, 224)
      文字: "这是什么?" → tokenizer → input_ids (1, 5)

════════════════ Step 1: 图像编码 (ViT, 冻结不训练) ════════════════

  (1, 3, 224, 224)
       │
       ▼  PatchEmbedding (Conv2d)
  (1, 196, 768)                 ← 196 个 patch, 每个 768 维
       │
       ▼  + [CLS] token
  (1, 197, 768)                 ← 196 patch + 1 个 [CLS]
       │
       ▼  + 可学习位置编码
  (1, 197, 768)
       │
       ▼  12 × ViTEncoderBlock (双向注意力, 无掩码)
  (1, 197, 768)                 ← 每个 patch 融合了全局视觉信息
       │
       │  forward_features() 只取这个输出, 不做分类头
       │

════════════════ Step 2: 投影到文本空间 ════════════════

  image_features: (1, 197, 768)  ← ViT 输出, 在视觉特征空间
       │
       ▼  ImageTextProjector
       │    Linear(768 → 768):  (1, 197, 768) → (1, 197, 768)
       │      把视觉768维映射到文本768维
       │      虽然维度相同, 但权重新学, 改变了语义空间
       │    GELU:  引入非线性
       │    Linear(768 → 768):  (1, 197, 768) → (1, 197, 768)
       │      第二层进一步调整映射
       ▼
  image_embeds: (1, 197, 768)    ← 现在在文本特征空间了!

════════════════ Step 3: 文本嵌入 ════════════════

  input_ids: (1, 5)              ← "这是什么?" 5 个 token
       │
       ▼  Token Embedding 查表
  (1, 5, 768)
       │
       ▼  × √768 ≈ 27.7         ← 缩放, 和纯语言模型一样
  text_embeds: (1, 5, 768)

════════════════ Step 4: 拼接 [视觉 | 文本] ════════════════

  image_embeds: (1, 197, 768)    ← 197 个视觉 token
  text_embeds:  (1,   5, 768)    ←   5 个文本 token
       │
       ▼  torch.cat([image_embeds, text_embeds], dim=1)
  combined: (1, 202, 768)        ← 197 + 5 = 202 个 token

  序列布局:
    位置 0~196:  [CLS] p1 p2 ... p196    ← 视觉 token
    位置 197~201: 这 是 什 么 ?           ← 文本 token

════════════════ Step 5: 位置编码 ════════════════

  combined: (1, 202, 768)
       │
       ▼  + SinusoidalPE(pos)    ← 正弦位置编码, 加一次
  (1, 202, 768)
       │
       │  位置0: 视觉[CLS] + PE[0]
       │  位置1: 视觉patch1 + PE[1]
       │  ...
       │  位置196: 视觉patch196 + PE[196]
       │  位置197: "这" + PE[197]   ← 注意: 文本从位置197开始!
       │  位置198: "是" + PE[198]
       │  ...

════════════════ Step 6: 因果掩码 ════════════════

  因果掩码 (202×202 下三角):

    [1 0 0 ... 0 0]   位置0 (视觉[CLS])   只看自己
    [1 1 0 ... 0 0]   位置1 (视觉patch1)  看位置0,1
    [1 1 1 ... 0 0]   ...
    [1 1 1 ... 1 0]   位置197 ("这")      看所有视觉token + 自己
    [1 1 1 ... 1 1]   位置198 ("是")      看所有视觉token + "这" + 自己
    ...

  ┌──────────────────────────────────────────────────────────────┐
  │  关键: 文本 token 能看到所有视觉 token!                      │
  │                                                              │
  │  "这" (位置197) 能看到位置0~197 (所有197个视觉token + 自己) │
  │    → 文本 token 可以"参考"图片内容                          │
  │                                                              │
  │  视觉 token 之间也是因果掩码 (不是双向!)                     │
  │    视觉[CLS]只能看自己, 视觉patch1能看[CLS]+自己            │
  │    → 但 ViT 已经做了双向注意力, 视觉token已经互相看过了     │
  │    → 到 Decoder 时每个视觉token已经包含了全局视觉信息        │
  │    → 因果掩码在这里对视觉部分影响不大                       │
  └──────────────────────────────────────────────────────────────┘

════════════════ Step 7: Decoder Blocks ════════════════

  (1, 202, 768)
       │
       ▼  10 × DecoderBlock (因果自注意力 + FFN)
       │    每层: norm → masked self-attn → 残差
       │          norm → FFN → 残差
       │
       │    注意力计算 (以位置198 "是" 为例):
       │      q_是 = w_q(x_是)           ← "是"的查询向量
       │      k = [k_CLS, k_p1, ..., k_p196, k_这, k_是]  ← 所有可见位置的键
       │      v = [v_CLS, v_p1, ..., v_p196, v_这, v_是]  ← 所有可见位置的值
       │
       │      scores = q_是 · k^T / √64   ← "是"对每个可见位置的分数
       │        → 对197个视觉token: 分数高 → "是"在参考图片内容
       │        → 对"这": 分数高 → "是"也在参考前一个文字
       │
       │      out_是 = Σ softmax(scores) × v ← 加权汇总
       │        = 视觉信息的加权 + 文字上下文的加权
       │
  (1, 202, 768)                ← 形状不变, 但每个token融合了图文信息

════════════════ Step 8: 输出投影 ════════════════

  (1, 202, 768)
       │
       ▼  LayerNorm
  (1, 202, 768)
       │
       ▼  Linear(768 → vocab_size)
  logits: (1, 202, vocab_size)  ← 每个位置对词表的预测分数

════════════════ Step 9: 计算损失 (只对文本部分!) ════════════════

  为什么只对文本部分算 loss?
    视觉token (位置0~196) 的 "下一个词预测" 没有意义
    → ViT 已经编码好了, 不需要从视觉token预测文字
    → 只关心: 给了图片+问题, 模型能不能生成正确答案

  取文本部分的 logits:
    text_logits = logits[:, 197:-1, :]    ← 位置197~200的预测
    text_targets = input_ids[:, 1:]        ← 位置198~201的目标 (shifted)

  loss = CrossEntropy(text_logits, text_targets, ignore_index=pad_token_id)

  数值示例:
    位置197 "这" 的 logits → 预测 "是"  ← 正确则 loss 小
    位置198 "是" 的 logits → 预测 "什"  ← 正确则 loss 小
    ...
    模型学习: 看了图片 + "这是什么?" → 应该输出 "猫"
```

### 6.5 推理时: 自回归生成 (逐步详解)

```
输入: 一张猫的图片 + 文字 "图里是什么?"

═════════════ 准备阶段 (只做一次!) ═════════════

  图片 → ViT → Projector → image_embeds (1, 197, 768)  ← 固定不动!
  后面每一步都复用这个 image_embeds, 不会重新算

═════════════ 第1步: 预测第一个字 ═════════════

  文字: "图里是什么?" → tokenizer → [12, 5, 89, 3, 45, 2]  (6个token)

  拼接: [image_embeds | text_embeds]
    位置0~196:  [CLS] p1 p2 ... p196      ← 197个视觉token
    位置197~202: 图  里 是 什 么 ?          ← 6个文本token
  combined: (1, 203, 768)

  → 位置编码 → 因果掩码 → 10层Decoder → logits (1, 203, vocab_size)

  取最后一个位置 (位置202, "?") 的 logits → softmax → 概率分布:
    "猫": 0.65    ← 最高! 模型看了图片, 认为图里是猫
    "狗": 0.08
    "鱼": 0.05
    ...
  采样 → "猫"

  此时生成: "图里是什么?猫"

═════════════ 第2步: 预测第二个字 ═════════════

  文字: "图里是什么?猫" → [12, 5, 89, 3, 45, 2, 37]  (7个token)
                           ↑ 之前6个 + 新生成的 "猫" (id=37)

  拼接: [image_embeds | text_embeds]
    位置0~196:  [CLS] p1 p2 ... p196      ← 视觉token (不变)
    位置197~203: 图  里 是 什 么 ? 猫       ← 多了一个"猫"
  combined: (1, 204, 768)

  → 位置编码 → 因果掩码 → 10层Decoder → logits (1, 204, vocab_size)

  取最后一个位置 (位置203, "猫") 的 logits → softmax:
    "在": 0.55    ← 模型认为"猫在"是最可能的延续
    "吃": 0.12
    ...
  采样 → "在"

  此时生成: "图里是什么?猫在"

═════════════ 第3步: 预测第三个字 ═════════════

  文字: "图里是什么?猫在" → 8个token

  拼接: [image_embeds | text_embeds]
    位置0~196:  视觉token (不变)
    位置197~204: 图 里 是 什 么 ? 猫 在
  combined: (1, 205, 768)

  → Decoder → 最后位置 logits → 采样 → "草"

  此时生成: "图里是什么?猫在草"

═════════════ 第4步 ═════════════

  → 采样 → "地"
  此时生成: "图里是什么?猫在草地"

═════════════ 第5步 ═════════════

  → 采样 → "上"
  此时生成: "图里是什么?猫在草地上"

═════════════ 第6步: 生成结束符 ═════════════

  → 采样 → <EOS> (id=2)

  检测到 <EOS>, 停止生成!

  最终输出: "猫在草地上"

═════════════ 关键机制总结 ═════════════

  1. image_embeds 只算一次, 每步复用 → 省计算
  2. 每步把所有已生成的文字重新编码, 拼到 image_embeds 后面
  3. 只看最后一个位置的 logits 预测下一个字
  4. 因果掩码保证: 最后一个位置已经"看到"了所有视觉token + 之前所有文字
  5. 生成 <EOS> 或达到 max_new_tokens 就停

  为什么每次都要重新跑整个 Decoder?
    因为每多一个字, 序列变了, 位置编码变了, 注意力权重也变了
    → 不能缓存之前的结果, 必须重新算
    → 这就是自回归生成慢的原因 (每步都要完整前向传播)
```

### 6.6 Projector 为什么需要?

```
ViT 输出: (1, 197, 768) — 图像特征空间
Decoder 期望: (1, T, 768) — 文本特征空间

虽然维度都是768, 但"语义空间"不同:
  ViT 的768维: 编码的是视觉信息 (边缘/纹理/颜色分布)
  Decoder的768维: 编码的是语言信息 (词义/语法)

Projector 的作用: 学习一个从视觉空间到文本空间的映射
  就像翻译官: 把"图像语言"翻译成"文字语言"

两层 MLP + GELU:
  Linear(768→768) → GELU → Linear(768→768)
  为什么两层? 单层线性可能不够, 两层+激活有非线性
  为什么用GELU不用ReLU? GELU在Transformer系列中效果更好

  维度流转:
    输入:  (1, 197, 768)   ← 视觉特征
    Linear1: (1, 197, 768) ← 第一层投影, 改变语义空间方向
    GELU:    (1, 197, 768) ← 非线性变换, 平滑的ReLU变体
    Linear2: (1, 197, 768) ← 第二层投影, 精细调整到文本空间
    输出:  (1, 197, 768)   ← 形状不变, 但语义从"视觉"变成"文本"
```

### 6.7 为什么冻结 ViT?

```
freeze_vit=True 时, ViT 参数不更新, 只训练 Projector + Decoder

原因:
  1. ViT 通常用海量数据 (ImageNet-21K, 14M图片) 预训练
     → 视觉特征已经很好, 不需要重新学
  2. 冻结 ViT = 少训很多参数 = 更快更省显存
  3. 避免灾难性遗忘: 如果微调ViT, 可能破坏原有视觉能力

什么情况下解冻?
  数据充足 + 需要ViT适应新领域 (如医学影像)
  解冻后学习率要很小, 避免破坏预训练权重
```

### 6.8 和纯 ViT / 纯语言模型的对比

```
┌──────────────┬──────────────────┬──────────────────┬──────────────────┐
│              │ 纯 ViT           │ 纯语言模型        │ VisionLanguageModel│
│              │ (分类)           │ (文本生成)        │ (看图说话)        │
├──────────────┼──────────────────┼──────────────────┼──────────────────┤
│ 输入         │ 图像             │ 文字              │ 图像 + 文字       │
│ 输出         │ 类别编号         │ 文字              │ 文字              │
│ 注意力       │ 双向, 无掩码     │ 因果掩码          │ 因果掩码          │
│ 位置编码     │ 可学习, 1次      │ 正弦/RoPE         │ 正弦, 1次         │

│              │                  │                   │ 下一词预测        │
│ 代表模型     │ ViT, DeiT       │ GPT, Llama        │ LLaVA, InternVL   │
└──────────────┴──────────────────┴──────────────────┴──────────────────┘
```
